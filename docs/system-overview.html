生成链路

全流程围绕一份符合规范的 Excel 功能清单展开。CLI 和 Web 最终都把请求落到同一条 Python 管线，由中间 Markdown、Excel 模板、AI 调用和结构化校验共同产出交付物。

0. 基础数据 读取功能清单，生成模块树与录入文档元数据。
1. FPA 按 profile / strategy / rule_set 生成 FPA 工作量评估和审核副本。
2. COSMIC 生成 COSMIC 模板、AI 填充稿、项目功能点拆分表和 CFP 汇总。
3. 需求清单 基于功能清单与 CFP 等中间结果写出需求清单 Excel。
4. 需求说明书 基于元数据和功能需求章节写出 DOCX 交付物。
5. 日志与下载 记录运行日志、AI prompt / response、源数据快照和 Web 会话产物。

管线入口是 ai_gen_reimbursement_docs.pipeline.run_pipeline， 支持 gen-all、gen-basedata、gen-fpa、 gen-cosmic、gen-list、gen-spec。

工程结构

仓库按生成核心、Web 后端、Vue 前端、配置模板、测试、文档和发布脚本分层。核心生成逻辑仍在 Python 包中，Web 主要负责会话、任务、配置和人工预览体验。

后端与 API

Web 后端由 web_app.server:app 装配静态前端和各类 API 路由。任务执行层复用核心生成管线，按本机模式或远程服务模式处理文件路径、上传文件和下载包。

路由组 代表接口 职责
tasks.py /api/run-local、/api/run-upload、/api/tasks、/api/fpa/options 启动生成任务、上传运行、任务列表、关闭/重跑/取消/继续，以及 FPA 预览模块。
config.py /api/web-config、/api/user/config、/api/web-config/files/* 配置读取、编辑、备份、恢复、校验、包导入导出和用户级密钥保存。
auth.py /api/auth/login、/api/auth/me、/api/admin/invites 远程模式账户注册、登录、退出、改密和管理员邀请管理。
artifacts.py /api/download/{session_id}、/api/ai-log/{session_id}、/api/sessions/{session_id}/fpa/debug-records 下载交付物、打开本地输出目录、查看 AI 调用日志和 FPA 调试记录。
history.py /api/history、/api/history/{run_id}/download 运行历史查询、单次记录详情、历史产物打开和下载。
system.py /api/health、/api/version、/api/license/status、/api/license/activate 系统健康、版本、运行模式、许可证状态、激活和提示音。

前端状态边界

stores/config.ts 管理工作模式和后端状态；stores/auth.ts 管理登录态、管理员状态和强制改密；stores/steps.ts、stores/log.ts、stores/toast.ts 分别承载生成步骤、日志和提示。

统一 API 封装在 src/lib/api.ts，将非 2xx 响应转换为 ApiError，并对后端不可用消息做归一化。

FPA 体系

FPA 是当前成熟度最高的生成域：支持多 profile、多执行策略、规则集继承、领域上下文、AI 与规则混合、审核副本和预览 audit。

Profile 底层 kind 默认策略 适用场景
strict_fpa strict_fpa ai_first + strict_fpa_rs 标准 FPA 复核，按数据功能和事务功能拆分，不生成界面开发/接口开发等开发工作项表达。
unified_ui unified_ui rules_first + unified_ui_rs 报账模板友好口径，同一三级模块默认合并一条界面开发行。
multi_ui unified_ui rules_first + multi_ui_rs 按独立页面、业务对象、业务流程或用户端拆分多条界面开发行。
ui_api_mapping ui_api_mapping ai_first + ui_api_mapping_rs 展示功能过程到界面开发和接口开发的映射关系。

审核副本

正式生成 FPA 时额外输出 FPA工作量评估-check.xlsx，包含 FPA 结果、覆盖审核、Warnings、规则命中详情和 AI 原始返回。正式 Excel 保持模板结构，审核字段进入副本。

术语约束

FPA 相关页面固定使用 新增/修改功能点、类型、计算依据归类、计算依据说明、生成方式。不得替换为功能点类型、类型判定依据、说明详情等同义词。

COSMIC 体系

COSMIC 当前仍是批处理生成链路，主要用于从模块树和元数据生成项目功能点拆分表。相关文档已明确：后续应优先稳定结构化草稿、校验器和正式输出状态，再扩展人工审阅页。

数据模型

DataMovement 与 CosmicItem

数据移动包含序号、子过程描述、移动类型、数据组、数据属性和复用度；功能过程包含模块路径、功能用户、触发事件、过程名和数据移动列表。

生成模块

cosmic_ai、cosmic_md、cosmic_writer

AI 生成数据移动链，Markdown 承载空白模板和填充稿，Excel writer 将扁平化行写入模板并处理合并、公式、下拉和批注。

当前风险

草稿与正式输出边界

现有逻辑对部分质量问题以 warning 处理，Markdown 解析格式敏感，CFP 口径依赖模板公式和 Excel 重算。文档建议后续引入 passed/review_required/blocked 状态。

授权与运行模式

系统支持本机模式和远程服务模式。授权体系和远程账户体系并行存在：授权控制产品使用，账户控制远程 Web 访问。

本机模式

浏览器操作本机服务，后端直接读取本地 xlsx，交付物写入本地目录，并支持打开输出文件夹。

远程服务模式

浏览器上传文件，后端在会话目录中执行任务，交付物通过 ZIP 下载。远程模式需要登录，并支持管理员邀请。

许可证

ai_gen_reimbursement_docs/licensing 负责激活、许可证文件、数据包和加密工具。Web 系统路由提供状态查询和激活接口。

维护关注点

以下不是缺陷清单，而是后续改造和验收时应保持显式的系统边界。

FPA 组合必须带指纹

FPA 结果质量由 profile × kind × strategy × rule_set × prompt × model 共同决定。稳定性报告和 audit trace 必须记录组合指纹，否则 warning 上升时无法判断是模型退化、规则集变更还是 profile 混搭问题。

COSMIC 需要区分草稿和正式交付

COSMIC 文档已指出：无 API Key、AI 失败、空数据移动、缺少触发事件或缺少 CFP 公式时，不应把目标 Excel 路径当成成功。后续宜用结构化校验报告和 passed/review_required/blocked 状态表达结果。

配置编辑需要保留审计和回滚

Web 配置中心已经提供备份、恢复、diff、文件级校验和配置包能力。后续新增配置项时，应同步更新配置服务、前端编辑页面、校验逻辑和测试。

预览页术语不能自由发挥

FPA 预览、审阅、修改页面必须遵循 docs/fpa/result-review-terminology.md。新增 COSMIC 或其他生成器审阅页前，应先确认业务域术语映射。

旧兼容路径可清理，但测试和文档必须同步

项目尚未上线，仓库规则允许优先采用清晰一致的实现并移除旧逻辑。每次调整已有行为时，应同步更新文档和测试，让当前目标行为可验收。
