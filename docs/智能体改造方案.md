# AI-Gen-Reimbursement-docs 智能体改造方案

## 一、项目现状

`ard` 是一个 AI 辅助的报账文档生成工具，核心流程：

```
功能清单 Excel → 解析模块树 → AI 填充模板 → 生成 4 类交付物
                                              ├── FPA 工作量评估.xlsx
                                              ├── COSMIC 功能点拆分表.xlsx
                                              ├── 项目需求清单.xlsx
                                              └── 项目需求说明书.docx
```

### 现有能力

| 维度 | 已具备 |
|------|--------|
| 输入 | Excel 功能清单解析、模块树提取 |
| 处理 | 多步流水线（按依赖自动编排）、LLM 模板填充、多套 FPA 口径 |
| 输出 | Excel/Word 文档生成、自定义模板 |
| 交互 | CLI（`ard` 命令）、Web UI（Vue + Python 后端） |
| 周边 | 配置管理、版本检查、运行历史、授权加密 |

---

## 二、差距分析

智能体核心六要素对照：

| 要素 | 现状 | 差距 |
|------|------|------|
| **自然语言交互** | CLI 参数 + Web UI 表单 | 无对话式入口，"我要生成 XX 项目报账"无法驱动 |
| **自主规划** | 固定流水线按依赖顺序 | 无法根据上下文动态调整策略（跳过某步、换口径） |
| **工具调用** | 文件读写、Excel/Word 操作 | 可扩展为 Agent Tool（查历史项目、发邮件、推 OA） |
| **记忆/上下文** | 无跨会话记忆 | 无法记住用户偏好、历史参数、常用模板 |
| **决策能力** | 模板驱动、规则明确 | 无法评审生成质量、自动修正、根据输入特征选策略 |
| **交互式澄清** | Web UI 有 `waiting_input` 状态 | 无法在信息模糊时主动追问、逐项确认 |

## 三、改造路线图（三层递进）

### 层次一：对话式入口 🟢 低改动

**目标**：在现有 CLI/Web UI 之上加自然语言对话层，用户说人话，Agent 翻译为现有参数执行。

```
用户："帮我生成 XX 项目的全套报账文档，功能清单在桌面"
         │
         ▼
    NL → 意图解析 → 参数映射 → 现有 ard CLI
         │
         ▼
    执行流水线 → 返回结果 + 产物下载链接
```

**改造内容**：

1. **新增 Chat 路由 / API**
   - `POST /api/chat` — 接收自然语言消息，返回 SSE 流式响应
   - 会话管理：每个会话绑定一个 `session_id`，维护消息历史

2. **意图解析 Prompt**
   - 将用户自然语言解析为结构化参数：
     ```json
     {
       "action": "gen_all | gen_fpa | gen_cosmic | gen_list | gen_spec",
       "source_file": "路径或描述",
       "options": { "output_dir": "...", "project_name": "...", "fpa_profile": "..." },
       "missing": ["需要确认的字段"]
     }
     ```
   - 信息不全时返回 `missing` 列表，触发追问

3. **前端 Chat 组件**
   - 消息列表 + 输入框 + 流式渲染
   - 产物卡片：生成完成后展示下载链接
   - 追问卡片：当 Agent 需要更多信息时，渲染选项按钮

4. **与现有 `pipeline.py` 对接**
   - Chat 层拿到结构化参数后，调用现有流水线函数
   - 进度通过 SSE 实时推送（已有 `tasks.py` 的 SSE 模式可复用）

**涉及文件**（预估）：

| 文件 | 改动 |
|------|------|
| `web_app/routes/chat.py` | **新增** Chat API 路由 |
| `web_app/services/chat_service.py` | **新增** 意图解析 + 对话管理 |
| `web_app/src/components/ChatPanel.vue` | **新增** Chat 前端组件 |
| `web_app/src/stores/chat.ts` | **新增** 会话状态管理 |
| `web_app/routes/tasks.py` | 微调，复用 SSE 进度推送 |
| `ai_gen_reimbursement_docs/llm_client.py` | 可能复用/扩展，新增 Chat 专用 prompt |

### 层次二：半自主 Agent 🟡 中改动

**目标**：Agent 不只是翻译指令，还要能主动思考、追问、自检。

```
用户："帮我准备 XX 项目的报账材料"
         │
         ▼
    Agent 分析功能清单 → 发现模块划分模糊 → 追问
         │
         ▼
    用户澄清 → Agent 确认策略 → 执行生成
         │
         ▼
    自检：FPA 表格是否完整？→ 发现遗漏 → 自动修正
         │
         ▼
    输出结果 + 质量报告
```

**改造内容**：

#### 2.1 主动追问引擎

在生成前增加"预检"阶段：

```
预检规则（可配置）：
  - 模块树层级是否 >= 2？
  - 功能点描述是否包含"新增/修改"？
  - 基准值公式是否已填写？
  - COSMIC 计数是否与模块树匹配？
  - 是否有空白 Sheet？
```

发现缺口时，Agent 不直接执行，而是生成追问并等待确认：

```
Agent："检测到以下问题：
       1. 模块'用户管理'下未填写功能点
       2. FPA 基准值公式为空，将使用默认公式 xxx
       请确认是否继续，或补充以上信息？"
```

实现方式：
- `PreflightCheck` 类，定义规则集，返回 `CheckResult[]`
- Chat 层在调用流水线前执行预检
- 前端渲染确认卡片

#### 2.2 质量自检

生成完成后自动检查：

```python
class QualityChecker:
    def check_fpa(self, xlsx_path) -> list[Issue]:
        """检查 FPA 表格完整性"""
    def check_cosmic(self, xlsx_path) -> list[Issue]:
        """检查 COSMIC 拆分合理性"""
    def check_list(self, xlsx_path) -> list[Issue]:
        """检查需求清单与源数据一致性"""
```

发现问题时：
- 低风险（格式偏差）→ 自动修复
- 中风险（数据存疑）→ 标记 + 建议
- 高风险（数据矛盾）→ 暂停 + 追问

#### 2.3 上下文记忆

```
用户级记忆（跨会话）：
  - 常用模板路径偏好
  - FPA 口径偏好（标准/简化）
  - 历史项目名称列表

项目级记忆（单次会话内）：
  - 已确认的参数
  - 被修正过的问题
  - 生成产物路径
```

实现方式：
- 轻量 SQLite（`~/.ard/memory.db`）存储键值对
- 会话内用内存字典，会话间用 SQLite
- 每次对话开始时加载用户偏好，注入到 LLM 上下文

**涉及文件**（预估）：

| 文件 | 改动 |
|------|------|
| `ai_gen_reimbursement_docs/preflight.py` | **新增** 预检规则引擎 |
| `ai_gen_reimbursement_docs/quality_check.py` | **新增** 质量自检 |
| `ai_gen_reimbursement_docs/memory.py` | **新增** 用户/会话记忆 |
| `web_app/services/chat_service.py` | 扩展，集成预检 + 自检 |
| `web_app/src/components/ConfirmCard.vue` | **新增** 确认卡片组件 |
| `web_app/src/components/QualityReport.vue` | **新增** 质量报告组件 |

### 层次三：全自主 Agent 🔴 高改动

**目标**：从"工具"变为"同事"——给一个目标，全流程自主完成。

```
用户："XX 项目下周要报账，帮我准备好全部材料"
         │
         ▼
    Agent 自主查找最新功能清单
         │
         ▼
    分析数据完整性 → 自主追问（邮件/IM）
         │
         ▼
    选择最佳 FPA 口径 → 生成全部文档
         │
         ▼
    自检 + 交叉验证 → 修正
         │
         ▼
    打包 → 推送到 OA/邮件 → 通知干系人
```

**改造内容**：

#### 3.1 工具扩展（MCP / Function Calling）

```
内置工具集：
  - search_files(pattern)     → 在项目目录搜索文件
  - read_excel(path)          → 读取 Excel 内容摘要
  - send_email(to, content)   → 发送通知邮件
  - push_oa(file)             → 推送到 OA 系统
  - query_history(project)    → 查询历史报账记录
  - write_doc(path, content)  → 写入文档
```

Agent 自主决定何时调用哪个工具，形成完整闭环。

#### 3.2 多轮自主决策循环

```
┌─────────────────────────────────────────┐
│  Think → Plan → Act → Observe → Loop    │
│                                         │
│  1. Think: 分析当前状态                 │
│  2. Plan:  决定下一步做什么             │
│  3. Act:   调用工具执行                 │
│  4. Observe: 检查结果，判断是否完成     │
│  5. Loop 直到目标达成或需要人工介入      │
└─────────────────────────────────────────┘
```

#### 3.3 人机协作模式

| 模式 | 说明 |
|------|------|
| **全自动** | Agent 自主决策，仅在最终交付前请求确认 |
| **半自动** | 关键节点暂停，等人工确认后继续 |
| **建议模式** | Agent 只做分析和建议，不直接操作文件 |

用户在对话中可随时切换："这个步骤让我确认一下" / "后面的全自动吧"。

#### 3.4 外部系统集成

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  邮件系统  │    │  OA 系统  │    │ 项目管理  │
│  (SMTP)  │    │  (API)   │    │  (API)   │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │
     └───────────────┼───────────────┘
                     │
              ┌──────┴──────┐
              │  ard Agent  │
              └─────────────┘
```

**涉及文件**（预估）：

| 文件 | 改动 |
|------|------|
| `ai_gen_reimbursement_docs/agent/` | **新增** 智能体核心模块目录 |
| `ai_gen_reimbursement_docs/agent/core.py` | Think-Plan-Act 循环 |
| `ai_gen_reimbursement_docs/agent/tools/` | 工具集（文件、邮件、OA） |
| `ai_gen_reimbursement_docs/agent/orchestrator.py` | 多步骤编排 |
| `web_app/routes/agent.py` | Agent API 端点 |
| `web_app/src/components/AgentConsole.vue` | Agent 控制台前端 |

## 四、推荐实施路径

```
第 1 阶段（2-3 周）          第 2 阶段（2-3 周）          第 3 阶段（按需）
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│ 层次一：对话入口  │ ──► │ 层次二：半自主    │ ──► │ 层次三：全自主    │
│                 │      │                 │      │                 │
│ · Chat API      │      │ · 预检引擎      │      │ · 工具扩展      │
│ · 意图解析      │      │ · 质量自检      │      │ · 人机协作      │
│ · Chat UI       │      │ · 上下文记忆    │      │ · 外部集成      │
│ · 流水线对接    │      │ · 主动追问      │      │ · 自主决策循环   │
└─────────────────┘      └─────────────────┘      └─────────────────┘
```

### 第 1 阶段验收标准

- [ ] 自然语言输入可正确路由到现有流水线
- [ ] 信息不全时能正确追问至少 3 类缺口（源文件、输出目录、项目名）
- [ ] SSE 流式进度可用
- [ ] 产物可在对话界面下载
- [ ] 现有 CLI 和 Web UI 功能不受影响

### 第 2 阶段验收标准

- [ ] 预检覆盖至少 5 条规则，误报率 < 20%
- [ ] 质量自检能发现 FPA 表格缺失行、COSMIC 计数不一致
- [ ] 跨会话记住用户偏好（模板路径、FPA 口径）
- [ ] 同一项目二次生成时自动复用上次参数

### 第 3 阶段验收标准

- [ ] Agent 可在无人工干预下完成"查找 Excel → 生成 → 打包"全流程
- [ ] 支持至少 1 个外部系统集成（邮件或 OA）
- [ ] 人机协作 3 种模式均可切换

## 五、风险与注意事项

| 风险 | 应对 |
|------|------|
| LLM 幻觉导致参数映射错误 | 意图解析后始终展示解析结果让用户确认（层次一）；增加 Schema 校验 |
| 预检规则过于严格导致频繁打断 | 规则分"阻断"和"提醒"两级，提醒级不打断流程 |
| 质量自检耗时过长 | 自检在生成后异步执行，不阻塞产物交付 |
| 跨会话记忆隐私问题 | 记忆全部本地存储（SQLite），不上传云端 |
| 外部系统集成稳定性 | 每个集成独立封装，失败时降级（跳过而非崩溃） |

## 六、技术选型建议

| 层次 | LLM 角色 | 需要的模型能力 |
|------|---------|---------------|
| 层次一 | 意图解析（分类 + 槽位填充） | 轻量模型即可，如 Haiku |
| 层次二 | 意图 + 预检判断 + 质量评审 | 中端模型，如 Sonnet |
| 层次三 | 多步推理 + 工具编排 | 强推理模型，如 Opus |

**渐进策略**：同一套代码支持多模型，简单任务走轻量模型降低成本，复杂推理自动升级模型。

## 七、与现有架构的关系

```
                    ┌──────────────────────┐
                    │     用户入口层        │
                    │  CLI  │  Web UI  │ Chat │  ← 层次一在此新增
                    └─────────┬────────────┘
                              │
                    ┌─────────▼────────────┐
                    │     智能体层          │  ← 层次二/三在此新增
                    │  ┌────────────────┐  │
                    │  │ 意图解析       │  │
                    │  │ 预检引擎       │  │
                    │  │ 质量自检       │  │
                    │  │ 上下文记忆     │  │
                    │  │ 工具编排       │  │
                    │  └───────┬────────┘  │
                    └──────────┼───────────┘
                               │
                    ┌──────────▼───────────┐
                    │     核心流水线        │  ← 现有，基本不改
                    │  pipeline.py         │
                    │  gen_fpa / gen_cosmic│
                    │  gen_list / gen_spec │
                    └──────────────────────┘
```

核心原则：**智能体层是薄封装，不侵入核心流水线**。现有 `pipeline.py` 和各种 `gen_*.py` 保持不变，智能体层通过调用现有函数来实现更高层的自主行为。
