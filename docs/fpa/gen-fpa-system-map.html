FPA generation architecture · current as of 2026-06-09

gen-fpa 从输入、profile、规则、AI、审计到稳定性验收的完整地图

这份页面把当前 `gen-fpa` 的方方面面整理成一张工程手册：它如何选择口径，如何组织 prompt，如何让规则与 AI 协作，如何确认模糊点，如何审计结果，以及哪些 profile 的 harness 已成熟、哪些仍需要补强。

4 初始化配置提供的 profile：strict_fpa、unified_ui、multi_ui、ui_api_mapping。
3 底层 kind：strict_fpa、unified_ui、ui_api_mapping；multi_ui 复用 unified_ui kind。
4 生成策略：rules_first、ai_first、rules_only、ai_only。
0 strict-real-model-recommended 连续复测后 warning / quality issue / blocking retry 均归零。

生成流水线

gen-fpa 的核心不是单次让模型输出表格，而是把输入规范化、profile 配置、规则兜底、AI 生成、后处理、审计和稳定性检查串成一条可追踪流水线。

1. 输入分组 从功能清单或预览请求中取得三级模块、模块描述、功能过程 processes。
2. 解析配置 确定 profile、kind、strategy、rule_set、core_rules、system/user prompt。
3. 构造 Payload 加入 module、processes、domain_context、process_facts、merge_review、type_judgement、agent_review。
4. 规则 / AI 生成 按 strategy 决定先跑规则还是先跑 AI；rules_only 跳过 AI，ai_only 禁止规则补行。
5. 后处理与审核 规范名称、类型、来源流程、说明结构，运行 validator 和 quality_review。
6. 输出与审计 生成正式 FPA、check 副本、audit trace、稳定性报告或预览 debug。

输出 Profile

profile 是用户看到的业务口径名称；kind 是代码行为类型；rule_set、prompt 和 strategy 决定具体执行细节。当前默认 profile 是 `strict_fpa`。

Profile Kind 默认策略 输出形态 适用场景
strict_fpa strict_fpa ai_first 标准 FPA 数据功能和事务功能：ILF / EIF / EI / EQ / EO。 标准 FPA 复核、功能点稳定计量、需要逻辑事务合并的场景。
unified_ui unified_ui rules_first 三级模块默认一条界面开发行，非界面业务动作补充逻辑处理、查询、导出等行。 报账模板友好的开发工作量表达，需要“一页 + 业务动作”口径。
multi_ui unified_ui rules_first 允许同一三级模块拆成多条界面开发行，每条保留拆分理由。 确有多个独立页面、对象、流程或用户端，需要拆分界面行。
ui_api_mapping ui_api_mapping rules_first 每个功能过程默认生成界面开发 EI + 接口开发 ILF；显式接口/服务调用另生成 ILF。 需要展示“功能过程到界面与接口”的映射关系。

执行策略

四种 strategy

策略含义典型用途
rules_first规则优先，AI 只补充或复核。unified_ui / multi_ui / ui_api_mapping 默认。
ai_firstAI 优先，规则补齐 AI 未覆盖的行。strict_fpa 默认。
rules_only仅规则生成，不调用 AI。本地烟测、无 API、稳定性基线。
ai_only仅 AI 生成，不做规则补行；仍做校验、后处理和 warning。测试模型独立表现。

ai_only 的边界

`ai_only` 不是完全绕过规则系统。它只禁止规则生成行和 `rules_fallback` 补行；rule_set 仍用于配置校验、AI 结果后处理、非法类型兜底、类型冲突 warning、审核追踪和 AI cache key。

合法 AI type 与规则建议冲突时，`ai_only` 保留 AI type 并记录 warning；AI 返回非法 type 时仍会用 profile/rule_set 推断出的合法类型兜底。

Prompt Payload

prompt 模板只允许并要求 `${core_rules}`、`${judgement_rules}`、`${payload_json}` 三类占位符。domain context 已在 payload_json 中，不是独立占位符。

payload_json 主要内容

  • `module`：client_type、l1、l2、l3、l3_desc。
  • `processes`：process_id、process_name、description、type。
  • `domain_context`：系统边界、内部数据组、外部数据组、外部服务等领域上下文。
  • `process_facts`：规则化业务事实抽取结果。
  • `merge_review`：合并边界建议。
  • `type_judgement`：类型建议。
  • `agent_review`：角色化审计契约。

规则系统

row_planning_rules

控制规则兜底行如何规划，例如 unified_ui 默认三级模块级界面行、每个 process 的非界面动作行、ui_api_mapping 的默认界面/接口行。

ui_row process_rows one_row_per_process

type_mapping_rules

直接类型映射，支持 EI/EQ/EO/ILF/EIF，适合项目级特例、业务对象或数据组边界。strict_fpa 中优先于普通事务关键词。

ILF EIF EI/EQ/EO

keyword_rules

事务动作关键词规则，常用于导出/报表 -> EO，查询/查看 -> EQ，新增/提交/导入 -> EI。

查询 导出 新增

coverage_rules

控制来源流程覆盖和数据功能补齐。strict_fpa 下 AI 缺少 ILF/EIF 数据功能时，可通过 rules_fallback 补齐。

external data rules

识别外部系统维护、本系统引用的数据组；普通外部服务调用不自动生成 EIF。

complexity matrices

standard_fpa 按 DET/RET/FTR 矩阵复算复杂度和功能点数；legacy_workload 则按 type weight 简化计算。

AI、后处理与质量审核

AI 输出规范化

non destructive first
  1. 解析 JSON rows，跳过非对象或缺名称行。
  2. 按 profile 规范化名称和类型。
  3. 补全来源流程、分类依据、说明结构和复杂度字段。
  4. 记录规则命中详情、warning、后处理警告。
  5. ai_first 可在 AI 缺行时使用规则补齐，但会保留审计痕迹。

质量审核链

retryable signals
  1. `validator` 检查查询误判 EI、普通服务误判 EIF、来源流程越界、说明结构缺失。
  2. `merge_review` 检查合并建议是否被最终 rows 采纳。
  3. `type_judgement` 检查高置信类型建议是否被偏离。
  4. `quality_review` 汇总 issue，并为 ai_first 提供一次重试反馈。
  5. 重试后仍有问题时保留结果并写入 warning/audit trace。

确认流

确认机制把“模型猜测”变成“用户确认的项目口径”。它不是自由聊天，而是结构化争议点确认。

auto

自动模式不暂停，不返回确认项。适合批量生成和无人值守路径。

cautious

审慎模式只在高风险 validator issue 上返回确认。当前 Web 默认使用该模式。

strict

严格确认模式除高风险问题外，也可返回说明结构等审阅项。适合验收或审计。

确认作用域

`current_run` 默认只影响本次生成；`project_profile` 会持久化为项目默认口径。正式批量生成中，模块产生确认项时会进入 waiting_input，前端提交 confirmed_decisions 后继续当前模块和后续流程。

Agent Review 骨架

当前“多 Agent”不是多个 LLM 调用，而是多角色确定性契约。它已经把接口和审计形态推进到位，但 strict_fpa 外的 profile 不应直接复用 strict 的语义作为硬约束。

当前角色

agent_review
  roles:
    business_fact_extractor -> process_facts
    fpa_type_judge -> type_judgement
    merge_boundary_reviewer -> merge_review
    quality_reviewer -> quality_review

下一步方向

不建议每个 profile 复制一套 Python 流程。更好的路线是通用 Agent Review 引擎 + profile contract。

build_agent_review(profile_contract, group, rows)

profile contract 描述 categories、judgement_rules、merge_rules、quality_checks、prompt_exposure、naming_policy。

Harness 成熟度

strict_fpa 的稳定性主线已阶段收口；其它 profile 有基础测试，但还没有同等级 golden + 真实模型趋势质量门。4 种 strategy 和不同 rule_set 可以混搭，但 harness 不能把所有组合都当成同等级保证，需要按组合分层。

Profile 当前 harness 成熟度 建议补强
strict_fpa golden fixtures、结构化断言、validator、确认流、agent_review、稳定性报告、真实模型 recommended 连续复测。 维护期基线 仅在模型、prompt、rule_set 或 fixture 变化时重新 fresh 抽样。
unified_ui 配置校验、prompt 渲染、规则兜底、部分 acceptance 测试。 基础可用 补 2-3 个 profile golden fixtures，定义漏界面、漏接口、重复计数等行为断言。
multi_ui 多界面同名行保留、review 元数据测试；主要依赖 prompt/rule_set。 偏薄 补拆分理由、独立页面/对象/流程/用户端的 fixture 和真实模型 smoke。
ui_api_mapping 默认界面/接口行、明确后端调用行、多接口行、重复默认行测试。 规则较清楚 补 AI 稳定性抽样和显式接口抽取过度/不足的断言。

组合分层

实际运行组合由 profile、kind、strategy、rule_set、prompt 和 model 共同决定。Harness 的目标不是穷举证明所有混搭都业务正确，而是定义推荐基线、继承扩展断言、阻止危险组合，并对实验组合提供可追踪审计。

等级 含义 当前示例 Harness 要求
certified 推荐组合，承诺业务口径稳定。 `strict_fpa + ai_first + strict_fpa_rs` golden fixtures、行为断言、真实模型抽样、质量门。
supported 允许使用，有基础规则和审计闭环。 `unified_ui + rules_first + unified_ui_rs`、`ui_api_mapping + rules_first + ui_api_mapping_rs` 配置校验、规则/AI smoke、schema 校验、source_process_ids 校验。
experimental 技术上可运行，但不承诺业务口径稳定。 跨口径 profile/rule_set 混搭、自定义模型或 prompt 试验。 防崩溃、清晰 warning、audit trace、生成来源可追踪。
invalid 语义明显不兼容或配置引用错误。 不存在的 profile、未知 kind、缺失 rule_set/core_rules/prompt。 直接报错，不回退默认组合。
规则集扩展推荐使用 `base harness + extension assertions`：继承 `strict_fpa_rs` 的客户规则集复用 strict 基线，只为新增外部数据组、特殊业务对象或合并边界补增量 fixture。

交付物与审计产物

正式 FPA 结果

用户可见的 FPA 工作量评估结果，遵循 result review terminology：新增/修改功能点、类型、计算依据归类、计算依据说明、生成方式。

Check 副本

`FPA工作量评估-check.xlsx` 记录 profile、kind、strategy、rule_set、来源流程、规则命中、拆分理由、warning。

Audit Trace

`fpa_audit_trace.json` 记录模块级生成来源、quality_review、agent_review、retry_trigger_source 和稳定性汇总。

稳定性报告

多 trace Markdown 对比报告和 Excel 稳定性 sheet，追踪 warning、retry、confirmation、quality issue 等趋势。

预览 Debug

Web FPA 预览返回 prompt、raw_response、parsed_rows、final_rows、quality_review 等调试信息。

确认记录

confirmed_decisions 可作为 current_run 或 project_profile 作用域进入下一轮 prompt，避免同一争议点反复摇摆。
