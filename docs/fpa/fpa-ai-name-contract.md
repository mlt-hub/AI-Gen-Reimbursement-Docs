# FPA AI name 字段契约

## 背景

FPA 输出表中的 `新增/修改功能点` 来自生成流程中的 `name` 字段。AI 适合判断功能点语义、类型和说明，但不适合单独承担稳定的字段格式契约。

在真实样例中，AI 可能返回多种 `name` 格式：

```text
地市后台-垂直行业营销-垂直行业管理-垂直行业管理-垂直行业数据组
【地市后台】垂直行业营销-装修管理-首页装修-行业首页装修数据
自定义页面列表查询
和乐业-垂直行业营销-签到管理-签到记录-查询签到记录
```

这些差异会让 FPA 工作量评估表 D 列看起来口径不一致，也会把 `子系统（模块）` 误混入功能点名称。

## 目标行为

所有 AI 生成的 FPA 行，输出到 `新增/修改功能点` 前必须统一为：

```text
【客户端类型】一级模块-二级模块-三级模块-功能点名称
```

前缀只能来自源功能清单解析后的三级模块上下文：

```text
client_type / l1 / l2 / l3
```

AI 可以决定最后一段 `功能点名称`，但不得决定或改写前缀。

## 职责边界

AI 负责：

- 识别数据功能和事务功能。
- 生成最后一段功能点名称。
- 判断 `类型`。
- 选择 `classification_basis_index`。
- 补充 `计算依据说明`。

代码负责：

- 从源功能清单取得标准前缀。
- 修正 AI 返回的短名称、错误前缀、缺失 `【】` 的路径。
- 将规范化过程写入 `后处理警告` 和 `规则命中详情`。
- 保证 Excel D 列格式稳定。

## 规范化规则

对 AI 返回的 `name` 执行后处理：

| AI 返回 | 输出 |
|---|---|
| `自定义页面列表查询` | `【地市后台】垂直行业营销-装修管理-自定义装修-自定义页面列表查询` |
| `和乐业-垂直行业营销-签到管理-签到记录-查询签到记录` | `【地市后台】垂直行业营销-签到管理-签到记录-查询签到记录` |
| `地市后台-垂直行业营销-垂直行业管理-垂直行业管理-垂直行业数据组` | `【地市后台】垂直行业营销-垂直行业管理-垂直行业管理-垂直行业数据组` |
| `【地市后台】垂直行业营销-装修管理-首页装修-行业首页装修数据` | 保持不变 |

## 验收标准

- FPA 主表和 check 表中的 `新增/修改功能点` 均使用统一前缀格式。
- AI 原始返回仍保留在调试和审计信息中，便于追溯。
- 类型判断不依赖强制补齐后的前缀，避免前缀文字干扰 FPA 类型推断。
- 若后处理修改了 `name`，必须记录 `postprocess.ai_name_prefix` 规则命中。

## 当前能力边界

当前实现强保证的是结构正确：

```text
新增/修改功能点 必须落在源功能清单中确定的三级模块前缀下。
```

也就是说，类似下面的前缀必须来自已解析出的三级模块上下文：

```text
【地市后台】垂直行业营销-装修管理-首页装修-
```

代码可以保证 AI 不会把 `和乐业`、短名称、缺少 `【】` 的路径或错误模块路径原样写入 D 列。

当前实现不强保证的是数据集语义一定最佳：

```text
【地市后台】垂直行业营销-装修管理-首页装修-行业首页装修数据
```

其中 `行业首页装修数据` 是否一定是最合适的逻辑数据组名称，仍属于 FPA 业务判断。系统会通过类型规则和复核 warning 暴露不确定项，但不会假装在缺少资产台账、数据模型或人工确认的情况下 100% 判断正确。

## 轻量校验建议

在不增加额外配置复杂度的前提下，可以保留一条轻量校验：

```text
当前模块生成的 FPA 行，其前缀必须等于当前三级模块标准前缀。
```

例如正在处理 `首页装修` 时，只允许输出：

```text
【地市后台】垂直行业营销-装修管理-首页装修-...
```

不允许混入：

```text
【地市后台】垂直行业营销-装修管理-自定义装修-...
```

如果发生偏移，代码应优先按当前 `group` 修正，并记录 `postprocess.ai_name_prefix`。

## 暂缓的设计

暂不引入以下能力：

- `data_set_catalog`
- `data_set_rules`
- 数据集归属新字段
- 数据集审核新页面
- 规则学习和自动沉淀
- 复杂的数据集合并、拆分、归并逻辑

原因是这些能力会明显增加 FPA 模块复杂度，也会给用户带来额外维护成本。当前阶段更适合保持边界清晰：

```text
代码保证结构正确
AI 提供业务判断
warning 暴露不确定项
人工审核最终结果
```

## 未来可能扩展

如果真实使用中反复出现稳定的数据集归属问题，再考虑引入轻量的数据集归属规则。

优先解决的问题包括：

- 外部系统数据归属，例如统一用户中心、CRM、ERP、供应商平台。
- 容易重复计数的数据组，例如用户信息、订单信息、配置数据、记录数据。
- 关系类数据，例如管理员关系、绑定关系、映射关系。
- 项目专有名词，例如行业首页装修数据、自定义页面信息、签到奖品配置。

未来可以增加的数据结构示例：

```yaml
data_set_rules:
  - match_keywords: ["系统用户", "统一用户中心"]
    data_set_name: "统一用户中心账号"
    type: EIF
    owner_system: "统一用户中心"
    independent_boundary: true
    reason: "外部系统维护，本系统只引用。"

  - match_keywords: ["垂直行业管理员", "管理员关系"]
    data_set_name: "垂直行业管理员关系"
    type: ILF
    owner_system: "和乐业"
    independent_boundary: true
    reason: "本系统独立维护行业与管理员的关联关系。"
```

未来如果实现，应遵循渐进原则：

1. 默认不要求用户维护完整数据集字典。
2. 系统先生成结果，只把冲突、不确定、疑似重复的数据组暴露给人工确认。
3. 用户确认后再沉淀少量高价值规则。
4. 规则只处理反复出现、业务价值明确的归属问题。

这样可以提高数据集归属准确率，同时避免把 FPA 模块过度设计成复杂的数据治理系统。
