# Gen-FPA Web UI 设计改进方案

日期：2026-06-02

## 背景

本方案基于对 gen-fpa Web UI 模块的 huashu-design 5 维评审结果形成。当前界面已经具备统一的企业操作台气质，能支撑生成任务、运行日志、FPA 预览和基础错误反馈，但距离“专业 FPA 审阅工作台”还有差距。

本阶段不追求营销化视觉，也不建议引入重型组件库。改进重点应放在信息层级、任务路径、FPA 审阅承载方式和移动端可用性上。

## 目标行为

完成后，gen-fpa Web UI 应满足以下目标：

- 用户进入首页后，能快速理解“选择输入、选择生成模式、开始生成”的主路径。
- FPA 预览页从“表格预览工具”升级为“可审阅、可解释、可诊断”的工作台。
- 桌面端保持左右分区的高效操作形态，移动端不只是堆叠，而是重组为适合窄屏的任务流。
- 错误、空态、离线状态能说明问题来源和下一步操作，不只暴露 HTTP 状态。
- 用户可见术语严格遵循 `docs/fpa/result-review-terminology.md`，FPA 页面固定使用 `新增/修改功能点`、`类型`、`计算依据归类`、`计算依据说明`、`生成方式`。

## 当前评审结论

### 5 维评分

| 维度 | 评分 | 判断 |
|---|---:|---|
| 哲学一致 | 7/10 | 克制企业操作台方向正确，适合本项目工具属性。 |
| 视觉层级 | 6/10 | 卡片、边框、标题统一，但主次权重接近，关键动作不够突出。 |
| 细节完成度 | 5.5/10 | 空态、错误态、折叠区、表格密度仍有粗糙感。 |
| 功能体验 | 6/10 | 主流程可用，但生成与预览关系、运行前校验、失败提示还不够顺。 |
| 创新/记忆点 | 4/10 | 稳定但泛化，缺少 FPA 审阅工具的专属表达。 |

### 主要硬伤

1. 顶部导航入口过满，窄屏下换成两行，挤占首屏空间。
2. 首页空闲态日志面板视觉重量过大，压过任务设置。
3. `打开 FPA 预览` 被放在高级选项区域，和真实使用路径不匹配。
4. FPA 预览页移动端只是纵向堆叠，没有按窄屏任务流重组。
5. FPA 结果表格依赖横向滚动，移动端审阅成本高。
6. 错误反馈过于工程化，例如 `请求失败 (500)`，缺少用户可执行提示。

## 分阶段实施方案

## 阶段 1：导航与首页主路径降噪

优先级：P0

### 目标

让用户进入首页后第一眼看到的是生成任务本身，而不是被导航、状态徽标和日志空态分散注意力。

### 修改建议

1. 顶部导航保留高频入口：
   - 一级保留：`生成`、`预览`、`历史`、`配置`
   - 弱化：`授权`
   - 移入开发/高级入口：`提示词调试`

2. 重新排列导航顺序：
   - 推荐顺序：`生成`、`预览`、`历史`、`配置`
   - 理由：FPA 预览是生成前后都可能使用的核心工作台，应比授权、调试更靠前。

3. 窄屏导航改为更紧凑的结构：
   - 方案 A：保留横向 wrap，但隐藏低频入口。
   - 方案 B：一级入口横排，低频入口放入 `更多`。
   - 推荐先做方案 A，改动更小。

4. 首页空闲态日志区减重：
   - 空闲态把深色日志面板高度降低，或改为浅色说明卡。
   - 任务开始后再切换为完整深色控制台。
   - `运行日志` 保留，但不要在空闲时占据页面最强视觉权重。

### 验收标准

- 1440px 下首页主视觉焦点是 `生成任务` 和主按钮。
- 375px 下顶部导航不超过两行，且不挤掉任务标题。
- 后端离线时，用户仍能清楚看到下一步是启动后端或填写输入。
- 空闲态日志区域不再压过任务设置。

### 风险

- 隐藏低频入口可能影响已有用户习惯。应保留可发现路径，例如 `配置` 内提供调试入口链接。

## 阶段 2：FPA 预览入口前置

优先级：P0

### 目标

把 FPA 预览从高级配置附属入口提升为主任务路径的一部分。

### 修改建议

1. 在首页任务设置区增加明确的次级操作：
   - 主按钮：`开始生成`
   - 次按钮：`预览 FPA 功能点`

2. 将当前高级选项块内的 `打开 FPA 预览` 移出或弱化。

3. 入口文案建议：
   - `预览 FPA 功能点`
   - 辅助说明：`先按当前输入和 FPA 方案生成可审阅的功能点估算。`

4. 如果输入未填写，点击预览入口后不要直接跳到空页面：
   - 可以跳转并保留清晰空态。
   - 更推荐在首页提示：`请先选择功能清单输入来源。`

### 验收标准

- 用户无需展开高级选项即可发现 FPA 预览。
- 预览入口和生成入口在视觉上有主次关系。
- 输入缺失时不出现空白页或工程错误。

## 阶段 3：FPA 预览页信息架构重组

优先级：P0

### 目标

让 FPA 预览页成为独立工作台，而不是“左侧输入表单 + 右侧表格”的简单拼装。

### 推荐结构

桌面端：

```text
----------------------------------------------------------+
| FPA 预览工作台                         FPA / COSMIC / SPEC |
+----------------------+-----------------------------------+
| 输入来源              | 模块选择 / 方案 / 策略 / 规则集      |
| - 功能清单路径         | [生成基础数据] [生成预览]             |
| - 输出目录             |-----------------------------------|
| - 高级选项             | 状态摘要 / 警告 / 审核覆盖率          |
|                      |-----------------------------------|
|                      | 审阅结果表格                         |
+----------------------+-----------------------------------+
```

移动端：

```text
FPA 预览工作台
[输入来源折叠卡]
[预览设置卡]
[生成基础数据] [生成预览]
[状态摘要]
[结果卡片列表]
```

### 修改建议

1. `PreviewLayout` 支持移动端重组：
   - 桌面端保持 `360px + 1fr`。
   - 768px 以下改为顺序流，但输入来源区域应可折叠。
   - 输入来源卡不应只显示一个标题后占用空间。

2. FPA 预览控制区拆成三层：
   - 输入来源：功能清单、输出目录、高级选项。
   - 预览参数：三级模块、FPA 方案、执行策略、规则集。
   - 动作区：生成基础数据、生成预览。

3. `生成基础数据` 和 `生成预览` 两个动作要有明确依赖关系：
   - 未生成基础数据时，`生成预览` 禁用并说明原因。
   - 基础数据生成成功后，自动选中第一个三级模块可以保留，但需要给出提示。

### 验收标准

- 375px 下用户不用滚动很久才能看到 `生成基础数据`。
- 1440px 下输入来源、预览设置和结果区域边界清晰。
- `生成预览` 禁用时有可理解的原因。

## 阶段 4：FPA 审阅结果桌面表格与移动卡片双形态

优先级：P0

### 目标

解决当前结果表格在移动端依赖横向滚动的问题，同时增强 FPA 审阅的专业感。

### 桌面端建议

保留表格，但优化密度和层级：

- 表头顺序保持：
  - `#`
  - `新增/修改功能点`
  - `类型`
  - `生成方式`
  - `计算依据归类`
- `计算依据说明` 继续放在行内详情或展开区域，不放进主表头。
- `类型` 使用稳定 badge，例如 `EI`、`EO`、`EQ`、`ILF`、`EIF`。
- `生成方式` 使用中性 badge，例如 `规则`、`AI`、`兜底`。
- `计算依据归类` 控制为短文本，长内容进入 `计算依据说明`。

### 移动端建议

768px 以下改为卡片列表：

```text
新增/修改功能点
<row.name>

类型: EI        生成方式: AI
计算依据归类
<short basis>

[展开计算依据说明]
```

卡片设计要求：

- 每张卡首行只强调 `新增/修改功能点`。
- `类型` 和 `生成方式` 作为元信息，不抢业务对象。
- `计算依据说明` 默认折叠，避免长文本淹没列表。
- 卡片之间间距应小而清楚，适合连续审阅。

### 可选交互

- 增加 `全部展开/全部收起`。
- 增加结果数量摘要：`共 N 个新增/修改功能点，AI 生成 M 个，规则生成 K 个`。
- 增加异常项高亮：缺失功能过程、兜底生成、AI 失败后兜底。

### 验收标准

- 375px 下不需要横向滚动也能完整审阅单条结果。
- 桌面端仍可快速扫表。
- 用户可见文案符合 FPA 审阅术语规范。

## 阶段 5：错误、空态和诊断提示优化

优先级：P1

### 目标

把工程错误转换成用户能理解并采取行动的反馈。

### 修改建议

1. FPA options 加载失败：
   - 当前类似：`请求失败 (500)`
   - 建议：`无法加载 FPA 方案。请确认后端服务已启动，并检查 FPA 配置是否可读取。`

2. 后端离线：
   - 当前已有 `后端未连接`，但应进一步说明影响范围。
   - 建议：`当前可查看界面，不能生成基础数据或预览结果。请启动后端服务后重试。`

3. 输入缺失：
   - 建议统一为：`请先选择功能清单输入来源。`
   - 避免在同一页出现多个相近空态重复提示。

4. 生成基础数据失败：
   - 展示失败原因。
   - 保留用户输入。
   - 按原因给出下一步，例如路径不存在、文件格式不正确、后端未连接。

5. 预览生成失败：
   - 区分 AI 失败后兜底、规则失败、后端异常。
   - 如果有 warning，不应只放在结果底部，应在结果摘要区也有提示。

### 验收标准

- 常见失败不需要用户打开浏览器控制台。
- 错误文案包含问题和下一步。
- API 失败不会清空用户已填输入。

## 阶段 6：高级选项和调试入口降噪

优先级：P1

### 目标

让普通用户路径更轻，同时保留高级用户所需能力。

### 修改建议

1. 高级选项拆分为两组：
   - AI 配置：API Key、接口地址、模型、最大 Token 数。
   - FPA 策略：FPA 方案、执行策略、规则集。

2. 在首页主任务里默认只展示和生成直接相关的字段：
   - 操作模式
   - 输入来源
   - 输出目录
   - FPA 方案摘要

3. `提示词调试` 从主导航降级：
   - 可放入 `配置` 页。
   - 或在高级选项中增加 `打开提示词调试`。

4. FPA 方案如果是影响结果的重要选项，不应完全藏在高级选项里：
   - 首页可展示当前方案摘要。
   - 预览页可直接修改方案。

### 验收标准

- 普通用户不用理解 API Key、模型、规则集也能完成生成。
- 高级用户仍能找到完整设置。
- 首页高级选项展开后结构清楚，不再是长字段列表。

## 阶段 7：视觉细节统一

优先级：P2

### 目标

在不大改设计系统的前提下提升精致度。

### 修改建议

1. 控制卡片阴影和边框层级：
   - 主要容器保留 `surface`。
   - 次级区块减少阴影，避免层层卡片。

2. 统一状态色语义：
   - 成功：生成完成、覆盖率正常。
   - 警告：后端离线、配置缺失、兜底生成。
   - 危险：任务失败、文件读取失败。

3. 优化日志空态：
   - 空态不必用大面积深色。
   - 运行中再切换到深色日志控制台。

4. 优化按钮层级：
   - 每个区域只保留一个主按钮。
   - 次按钮使用 `btn-secondary`。
   - 低风险链接使用 `btn-quiet` 或文字链接。

5. 表单密度：
   - 桌面端可以保持紧凑。
   - 移动端字段间距略增，避免误触。

### 验收标准

- 页面整体仍保持克制企业工具气质。
- 不出现多个同权重主按钮。
- 空态、错误态、完成态风格一致。

实施后至少检查以下视口：

| 宽度 | 页面 | 检查点 |

## 不建议本轮做的事

- 不建议引入大型 UI 组件库。
- 不建议把首页改成营销式欢迎页。
- 不建议重写整个前端路由。
- 不建议在没有测试保护时大规模拆组件。
- 不建议创造新 FPA 术语替代既有术语规范。
- 不建议为了移动端完全删除桌面表格，桌面审阅效率仍然需要保留。

## 最小可执行下一步

如果只做第一步，建议优先完成：

1. 顶部导航降噪。
2. 首页前置 `预览 FPA 功能点`。
3. FPA 预览页移动端输入区可折叠。
4. FPA 结果移动端卡片展示。

这四项能最大幅度改善美观度、交互体验和响应式硬伤，且不需要改变后端接口。
