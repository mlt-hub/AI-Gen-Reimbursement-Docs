# 当前 Web UI 评审与改进建议

> 评审日期：2026-06-08  
> 评审范围：当前 Vue Web UI 的主要工作台页面、配置页、FPA 预览页、历史页、授权页、提示词调试页  
> 评审方式：启动 Vite 前端后，通过 Playwright 观察关键路由在桌面与窄屏视口下的真实渲染和页面文本  
> 本文性质：方案文档，不包含代码修改

## 结论摘要

当前 Web UI 的能力覆盖已经比较完整：生成入口、FPA 预览、配置管理、历史记录、授权、提示词调试都有明确页面，FPA 预览页也已经遵守结果审阅术语规范。

主要问题不是“界面不好看”，而是信息架构和操作层级还没有收拢：

- 首页首屏把主任务、执行监控、低频参数放在接近同等的视觉权重下，用户很难一眼判断当前最重要的动作是什么。
- 低频 FPA 参数过早暴露，占据了生成任务的主要视线。
- 后端未连接时，多个页面和多个模块重复出现 `请求失败 (500)`，错误态显得嘈杂。
- 配置页内容很全，但所有配置分区几乎同权铺开，更像长表单，不像可扫描、可定位的配置中心。
- FPA 预览页的术语正确，但局部卡片嵌套和空状态可以更紧凑，让审阅行为更靠近结果表格本身。

建议优先做“结构收口”，而不是先做大范围视觉换肤。

| 提示词调试页 | `/prompt-debug` | 双栏调试结构、空状态 |

### 响应式观察

观察过的视口包括：

| 视口 | 结论 |
|---|---|
| `390 x 844` | 未发现横向溢出，内容会纵向堆叠 |
| `768 x 1024` | 未发现横向溢出，表单宽度可用 |

响应式基础是可接受的，短期不需要把重点放在断点重写上。更值得先改的是页面层级、信息密度和错误呈现。

## 当前优点

### 1. 业务功能覆盖完整

当前路由已经覆盖 Web UI 的关键工作：

- 生成任务：`web_app/src/views/Home.vue`
- FPA 预览：`web_app/src/views/FpaPreviewPage.vue`、`web_app/src/components/FpaPreview.vue`
- 配置管理：`web_app/src/views/Config.vue`
- 历史记录：`web_app/src/views/History.vue`
- 授权：`web_app/src/views/License.vue`
- 提示词调试：`web_app/src/views/PromptDebug.vue`

这说明改进重点应是“让现有能力更易用”，不是推倒重做。

### 2. FPA 审阅术语基本正确

FPA 预览页已经使用以下规范术语：

- `新增/修改功能点`
- `类型`
- `生成方式`
- `计算依据归类`
- `计算依据说明`

这与 `docs/fpa/result-review-terminology.md` 保持一致。后续改 UI 时必须继续保留这些词，不要改回 `功能点类型`、`类型判定依据`、`功能点说明`、`说明详情` 等替代表述。

### 3. 页面已有基础设计系统

当前界面已经有统一的 CSS 变量、按钮样式、表单控件、surface 卡片、状态色和侧边导航。后续不需要引入大型 UI 组件库，也不应该用另一套视觉语言重写。

建议继续沿用现有变量和 Tailwind 组织方式，只调整布局结构、组件层级和状态表达。

## 核心问题

## P0：首页主任务路径不够清晰

### 现象

首页当前结构大致是：

1. `任务设置`
2. `执行监控`
3. `低频任务设置`

其中 `任务设置` 里又包含：

- 后端服务未连接提示
- 操作模式
- 功能清单路径
- FPA 预览入口
- 开始生成
- 任务状态

在桌面首屏中，`开始生成` 按钮横向铺开，位置靠近 `执行监控` 和 `低频任务设置` 的边界，视觉上像插在两个卡片之间。用户会看到很多可操作项，但不容易立刻判断“我现在要先做什么”。

### 影响

- 新用户会被 FPA 策略、运行参数、执行监控、提示词调试等低频信息分散注意力。
- `开始生成` 是主操作，但没有和输入路径形成足够紧密的行动区。
- 后端未连接时，用户看到禁用按钮和错误信息，但缺少一个清晰的下一步。

### 建议目标行为

首页首屏只强调一个主流程：

```text
选择生成内容 -> 填写功能清单路径 -> 开始生成 / 预览 FPA
```

执行监控只有在任务启动后才获得更高权重。低频参数默认收起，只在用户明确需要调整 FPA 方案、策略、规则集或输出目录时展开。

### 建议改法

1. 将首页改成左右两栏或上下两段的工作台结构：

| 区域 | 默认状态 | 说明 |
|---|---|---|
| 主操作区 | 展开 | 操作模式、功能清单路径、主按钮、FPA 预览入口 |
| 运行状态区 | 空状态轻量展示 | 未启动时只显示“等待任务启动”；运行后展开阶段进展 |
| 高级参数区 | 默认折叠 | 项目名称、输出目录、FPA 方案、执行策略、规则集、生成模式、清理旧输出 |

2. `开始生成` 应紧跟输入区，不要跨越整个页面压在区块之间。

3. `任务状态` 可以移动到运行状态区，不必在输入区重复显示。

4. `预览 FPA 功能点` 和 `开始生成` 要形成并列但有主次的动作：

| 动作 | 样式建议 | 说明 |
|---|---|---|
| 开始生成 | 主按钮 | 默认主路径 |
| 预览 FPA 功能点 | 次按钮或轻量链接按钮 | 审阅型路径，不应抢主按钮权重 |

### 涉及文件

- `web_app/src/views/Home.vue`
- `web_app/src/components/ConfigPanel.vue`
- `web_app/src/components/ActionBar.vue`
- `web_app/src/components/GenerationProgress.vue`
- `web_app/src/components/run/FpaRunSettingsSection.vue`

## P0：后端未连接错误态重复且噪声大

### 现象

当前只启动前端、不启动后端时，多个页面会显示：

- `后端未连接`
- `后端服务未连接`
- `请求失败 (500)`
- `无法加载 FPA 方案配置`
- `请确认后端服务已启动...`

配置页尤其明显，几乎每个配置卡片都显示一次 `请求失败 (500)`。

### 影响

- 用户会感觉系统整体坏掉，而不是“当前只是后端未启动”。
- 重复错误抢占了正常信息架构。
- `500` 是工程错误码，不是适合主界面反复展示的用户语言。

### 建议目标行为

后端不可用时，只保留一个强全局状态，局部模块进入安静的不可用态。

推荐表达：

```text
后端未连接
当前只能查看界面。启动后端服务后可运行任务、读取配置和生成预览。
```

局部模块只显示：

```text
等待后端连接后加载
```

不在每个卡片里重复显示 `请求失败 (500)`。

### 建议改法

1. 在 `AppShell` 或顶部状态区保留唯一的全局后端状态。

2. `Config.vue` 的配置区块在后端离线时显示统一空状态：

| 当前显示 | 建议显示 |
|---|---|
| 每块 `请求失败 (500)` | 页面顶部一个离线说明，配置区块显示“等待后端连接后加载” |

3. `FpaPreview.vue` 中 `无法加载 FPA 方案配置` 可以保留，但要降低视觉强度，并明确页面会使用内置默认方案。

4. `normalizeApiError` 可以继续保留原始错误，但只在排错详情、toast 或开发调试区域显示。

### 涉及文件

- `web_app/src/App.vue`
- `web_app/src/components/layout/AppShell.vue`
- `web_app/src/views/Config.vue`
- `web_app/src/components/FpaPreview.vue`
- `web_app/src/lib/api.ts`

## P1：配置页需要从长表单改成配置中心

### 现象

配置页当前从上到下铺开：

- 环境诊断
- AI 配置
- Web 与运行配置
- FPA 策略
- FPA 判定规则
- 业务规则
- 领域上下文
- Prompt 配置
- 模板配置
- 高级配置
- 配置备份
- 配置导入导出

内容完整，但视觉层级几乎相同，滚动距离长，用户要找某个配置时主要靠阅读和滚动。

### 影响

- 配置能力越多，页面越像“全部字段倾倒出来”。
- 高频配置和管理员级高级配置混在一起。
- 后端错误态下，页面被重复失败信息占满。

### 建议目标行为

配置页应该像配置中心：

```text
环境状态
常用配置
FPA 配置
模板与交付物
高级维护
备份与迁移
```

用户应能快速定位自己要改的是“AI Key / 默认输出 / FPA 方案 / 模板 / YAML 高级文件”中的哪一类。

### 建议改法

1. 页面顶部保留 `环境诊断`，但压缩为横向摘要条：

| 项 | 建议 |
|---|---|
| 后端连接 | 状态 badge |
| 工作模式 | 状态 badge |
| 模板目录 | 状态 badge |
| 配置接口 | 状态 badge |
| 最近检查 | 小字展示 |

2. 引入配置分区导航，推荐用 tabs 或左侧锚点：

| 分区 | 包含内容 |
|---|---|
| 常用 | AI 配置、运行默认值 |
| FPA | 方案与规则集、计算依据归类判定原则、领域上下文 |
| 模板 | 输出模板、下载模板、自定义输出模板 |
| 高级 | YAML / JSON 配置文件、Prompt 配置、业务规则 |
| 维护 | 备份恢复、导入导出 |

3. 高频表单保持展开，低频文本编辑、YAML、备份恢复默认折叠。

4. 每个分区只保留一个保存入口，避免多个 `刷新`、`保存` 按钮在同一视野里竞争。

### 涉及文件

- `web_app/src/views/Config.vue`
- `web_app/src/components/TemplateUpload.vue`
- `web_app/src/components/TemplateDownload.vue`

## P1：FPA 预览页应更像审阅工具

### 现象

FPA 预览页当前输入设置在左侧，右侧选择三级模块、FPA 方案、执行策略、规则集，然后生成基础数据和生成预览。页面空状态清晰，但右侧大面积空白较多，且 FPA 设置在左侧低频设置和右侧工具栏中都有出现。

### 影响

- 用户可能不确定应该先改左侧低频设置，还是右侧 FPA 方案。
- 预览结果尚未生成时，页面显得空而重。
- 结果出来后，如果每行说明都展开到表格中，审阅密度可能不足。

### 建议目标行为

FPA 预览页应该围绕“审阅结果”组织，而不是围绕“生成参数”组织。

推荐流程：

```text
选择输入 -> 生成基础数据 -> 选择三级模块 -> 生成预览 -> 审阅/确认/查看依据
```

### 建议改法

1. 左侧只保留输入来源和基础数据状态。

2. FPA 方案、执行策略、规则集合并到右侧顶部工具栏，不在左侧低频设置中重复出现。

3. 生成基础数据后，三级模块选择区域应显示模块数量、当前选择、功能过程数，让用户知道选择的审阅范围。

4. 预览结果区采用更紧凑的表格优先结构：

| 表头顺序 | 必须保留 |
|---|---|
| `#` | 是 |
| `新增/修改功能点` | 是 |
| `类型` | 是 |
| `生成方式` | 是 |
| `计算依据归类` | 是 |

5. `计算依据说明` 保持行内详情，不集中放到底部。

6. 需要人工确认的问题应放在结果上方的“待确认”区，并清楚区分：

| 类型 | 展示方式 |
|---|---|
| 当前运行确认 | 轻量确认面板 |
| 项目级经验沉淀 | 后续如果支持持久化，再提供明确的项目级保存动作 |

### 涉及文件

- `web_app/src/views/FpaPreviewPage.vue`
- `web_app/src/components/FpaPreview.vue`
- `web_app/src/components/run/FpaRunSettingsSection.vue`
- `web_app/src/composables/useFpaOptions.ts`

## P1：导航与页面命名可以更贴近任务

### 现象

当前侧边栏分为：

- 工作台：生成、预览、历史、配置
- 工具：授权、提示词调试

结构可用，但“预览”作为独立入口时，用户不一定知道这里主要是 FPA 预览；“提示词调试”作为普通工具入口对业务用户可能偏工程化。

### 建议改法

1. 如果短期只有 FPA 预览成熟，侧边栏可显示 `FPA 预览`，进入后再保留 `FPA / COSMIC / SPEC` tabs。

2. `提示词调试` 可以保留在工具区，但对远程普通用户隐藏或放入高级工具。

3. `授权` 对本机用户是维护动作，不应该和日常工作台竞争注意力。

### 涉及文件

- `web_app/src/router/index.ts`
- `web_app/src/components/layout/AppShell.vue`

## P2：视觉系统可以更紧凑，但不建议大换肤

### 现象

当前整体视觉是浅色、低饱和、圆角卡片、浅青强调色。这个方向适合内部工具，但存在两个问题：

- 卡片太多时，页面像一串同权容器。
- 主色用于导航、按钮、预览入口、状态等多个用途，区分度有限。

### 建议目标

保持“安静、可靠、工具化”的基调，减少装饰感，让用户更快扫描任务状态和关键动作。

### 建议改法

1. 卡片只用于真正需要成组的内容，不把每个页面区块都做成同重量 surface。

2. 主按钮、导航选中、提示区使用不同强度，不都依赖同一种浅青背景。

3. 错误、警告、不可用状态减少大面积填色，优先使用轻量状态条和 badge。

4. 表单控件高度、label 间距、分组标题统一收紧，提升操作型页面的扫描效率。

## 推荐实施顺序

### 第一阶段：首页主流程收口

目标：让用户进入首页后立刻知道怎么开始。

修改范围：

- `Home.vue`
- `ConfigPanel.vue`
- `ActionBar.vue`
- `FpaRunSettingsSection.vue`

验收标准：

- 首屏只出现一个主操作路径。
- `开始生成` 不再压在卡片边界或低频设置上方。
- 低频任务设置默认折叠。
- 后端未连接时，主按钮禁用原因清楚，但不重复刷多个错误。

风险：

- 需要确认 `FpaRunSettingsSection` 是否同时被首页和 FPA 预览页复用，避免改首页时影响预览页。

### 第二阶段：统一后端离线错误态

目标：减少 `请求失败 (500)` 噪声，让离线状态更像系统状态而不是页面报错。

修改范围：

- `App.vue`
- `AppShell.vue`
- `Config.vue`
- `FpaPreview.vue`
- `lib/api.ts`

验收标准：

- 后端离线时，侧边栏或顶部只显示一个明确状态。
- 配置页不再每个区块都显示 `请求失败 (500)`。
- 原始错误信息仍可在排错详情或开发调试区域看到。

风险：

- 需要避免吞掉真实后端异常。建议保留开发调试信息，但改变默认用户可见文案。

### 第三阶段：配置页分区化

目标：把长配置页改成可定位的配置中心。

修改范围：

- `Config.vue`
- 必要时抽取配置分区组件

验收标准：

- 页面顶部有环境摘要。
- 常用配置、FPA、模板、高级、维护分区明确。
- 用户不需要滚动很久才能找到 AI Key、FPA 方案、模板配置。
- 后端离线时显示统一空状态。

风险：

- `Config.vue` 当前文件较大，直接改容易扩大 diff。建议先做布局层拆分，再迁移具体表单。

### 第四阶段：FPA 预览审阅体验优化

目标：让 FPA 预览页更像审阅工具。

修改范围：

- `FpaPreviewPage.vue`
- `FpaPreview.vue`
- `FpaRunSettingsSection.vue`

验收标准：

- FPA 参数只出现一处，不在左右两边重复。
- 生成基础数据、选择三级模块、生成预览的步骤更明显。
- 表格字段继续使用规范术语。
- `计算依据说明` 保持行级详情。

风险：

- FPA 审阅术语不能漂移。
- 确认流相关文案要区分当前运行确认和项目级沉淀。

## 验收清单

### 视觉与布局

- 桌面 `1440 x 1000`：首页首屏主任务路径清楚。
- 桌面 `1280 x 720`：主按钮不压在上下区块之间。
- 平板 `768 x 1024`：无横向滚动，低频设置折叠可用。
- 手机 `390 x 844`：无横向滚动，按钮文字不溢出。

### 业务术语

- FPA 页面继续使用 `新增/修改功能点`、`类型`、`计算依据归类`、`计算依据说明`、`生成方式`。
- 不出现 `功能点类型`、`类型判定依据`、`功能点说明`、`说明详情` 等替代表述。

### 错误态

- 后端离线时只有一个主状态提示。
- 配置区块不重复刷 `请求失败 (500)`。
- 真实 API 错误仍可追踪。

### 任务流

- 未启动任务时，执行监控是轻量空状态。
- 任务运行后，阶段进展和交付物操作显著可见。
- 完成后，打开交付物目录或下载 ZIP 是最突出动作。

## 不建议做的事

- 不建议先引入新的 UI 组件库。当前问题主要是信息架构，不是组件能力不足。
- 不建议做营销式首页。这个系统是工作台，不需要 hero 区、宣传文案或大面积装饰图。
- 不建议把所有设置都放进弹窗。低频设置可以折叠，但仍应可追踪、可复制、可排错。
- 不建议为了减少错误文案而吞掉错误。应该改变用户默认看到的层级，同时保留排错信息。

## 下一步建议

如果按最小可交付推进，建议先实施第一阶段：

1. 首页主操作区收口。
2. 低频 FPA 设置默认折叠。
3. `开始生成` 与 `预览 FPA 功能点` 重新分配主次。
4. 后端离线时首页只保留一个明确提示。

这一步改动范围较小，但会显著改善用户第一次进入页面的判断成本。
