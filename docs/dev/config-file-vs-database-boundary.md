# 配置文件与数据库边界建议

## 背景

当前项目同时包含命令行生成管线和 Web 管理界面，已有两类持久化方式：

- 配置文件：`~/.ai-gen-reimbursement-docs` 下的 `.env`、`system_config.yaml`、`business_rules.yaml`、`fpa_config.yaml`、`fpa_judgement_rules.yaml`、`domain_context.json` 等。
- SQLite 数据库：用户认证、持久登录会话、邀请码、运行历史等。

本系统尚未上线，后续调整不需要保留旧版本兼容路径。边界选择应优先服务于清晰、一致、可维护和可验收。

## 判断原则

适合放配置文件的数据，通常具备以下特征：

- 低频修改，更多是部署、规则、模板或口径配置。
- 需要人工阅读、审查、复制、备份、导入导出。
- 需要通过 Git 或文件 diff 跟踪差异。
- 内容结构偏声明式，查询需求简单。
- 与模板、Prompt、领域规则等发布物强绑定。

适合放数据库的数据，通常具备以下特征：

- 运行期频繁产生或更新。
- 多用户共享，需要权限隔离、并发写入或事务一致性。
- 需要按用户、项目、时间、状态等条件查询。
- 需要审计、追溯、分页、统计或后台管理。
- 属于业务过程记录，而不是系统规则本身。

## 建议继续使用配置文件

### 运行与部署配置

- `.env`
  - API Key 加密值、模型名、模型地址等本机或用户级运行参数。
  - 敏感值应继续通过加密值和脱敏视图暴露，不应把明文写入数据库或审计记录。
- `system_config.yaml`
  - 默认工作模式、项目名、FPA profile、FPA strategy、远程下载保留期等低频配置。

### 业务规则与生成口径

- `business_rules.yaml`
  - 通用业务规则。
- `fpa_config.yaml`
  - FPA 计算、补齐、调整值、profile 相关配置。
- `fpa_judgement_rules.yaml`
  - FPA 类型判定规则、关键词规则、冲突处理规则。
- `domain_context.json`
  - 领域上下文、行业词汇、知识背景。
- `ai_system_prompts_config.yaml`
  - Prompt 模板、系统提示词和生成上下文模板。

这类文件目前更需要可读、可 diff、可备份和可导入导出。除非后续明确要做规则启停、灰度发布、项目级绑定、A/B 测试或后台增删改查，否则不建议过早数据库化。

### 模板与 Manifest

- 输出模板文件。
- 模板旁路或同目录 manifest。
- 模板能力声明、占位符、锚点、校验规则。

模板 manifest 与具体模板文件强绑定，继续文件化更清晰。数据库可以保存模板登记和使用记录，但不宜替代 manifest 本身。

### 内置默认值与代码口径

- 内置 profile、默认策略、常量、默认兜底逻辑。
- 例如严格 FPA profile 的代码级行为。

这类内容属于程序能力，不是运行期业务数据。应放代码或随包默认配置中，通过测试保证行为稳定。

## 建议使用数据库

### 用户、权限与登录状态

当前 `users.db` 的方向是合理的，应继续保持：

- 用户。
- 角色。
- 密码哈希和 salt。
- 邀请码。
- 持久登录 session。
- 禁用状态和首次修改密码状态。

这些数据涉及权限、并发和安全审计，不适合配置文件。

### 运行历史与任务记录

当前 `run_history.sqlite3` 的方向是合理的，应继续加强：

- 任务 ID。
- 执行来源。
- 本机或远程模式。
- 所属用户。
- 任务状态。
- 输入文件名。
- 输出目录或下载包。
- 生成产物列表。
- 错误信息。
- 执行配置快照。
- 创建、开始、完成和更新时间。

运行历史需要分页、过滤、按用户隔离和状态查询，应放数据库。文件系统只负责保存真实产物。

### Web 任务状态与进度事件

当前 `SessionManager` 主要保存内存态，适合短生命周期交互，但不适合作为可靠持久化来源。

建议后续将以下数据数据库化：

- 远程任务当前状态。
- 任务取消标记。
- 步骤进度。
- 等待用户输入状态。
- 产物事件。
- 失败原因。

这样服务重启后仍可展示历史状态，并能支撑多用户远程场景。

### 配置变更审计

当前配置审计使用 `audit/config_changes.jsonl`，单机开发期可接受。

如果要在 Web UI 中按条件查询审计记录，建议迁入数据库：

- 操作人。
- 目标配置作用域。
- 变更文件。
- 变更字段。
- 操作结果。
- 时间戳。

审计记录应保留脱敏字段，不记录 API Key 明文或可逆片段。

### 生成结果审阅与人工修改记录

以下属于业务过程数据，应使用数据库：

- FPA 生成结果审阅记录。
- 用户对生成结果的修改。
- 确认、驳回、重新生成记录。
- AI 建议与人工最终值的差异。
- 项目级复用的确认结论。

这些数据需要按项目、用户、时间和状态查询，不适合长期放在 JSON 文件里。

### 项目、工单与上传文件元数据

如果 Web 端开始管理多个项目，建议数据库保存：

- 项目。
- 工单。
- 上传文件元数据。
- 文件所属用户。
- 文件来源。
- 生成产物索引。
- 产物过期时间。

实际文件仍保存在文件系统或对象存储，数据库只保存元数据和索引。

## 灰区与演进建议

### 用户个人配置

当前远程模式下用户个人配置位于 `users/<username>/` 目录，开发期可以保留。

如果后续需要以下能力，应考虑迁入数据库：

- 管理员统一查看用户配置。
- 配置版本历史。
- 配置审批。
- 配置差异查询。
- 用户级配置继承和覆盖关系可视化。

### FPA 项目级确认决策

当前 `fpa_project_profile.json` 适合单机或轻量项目级复用。

如果后续要多人协同、跨项目复用、查询确认来源或审计修改记录，应迁入数据库。

### 业务规则和 Prompt

业务规则和 Prompt 当前应优先保持文件化。

当出现以下需求时，可采用“数据库存版本元数据，文件或文本字段存正文”的混合方案：

- 按项目绑定规则版本。
- 规则启停。
- Prompt A/B 测试。
- 规则发布审批。
- 生成结果追溯到具体规则版本。

## 当前推荐边界

短期保持文件化：

- `.env`
- `system_config.yaml`
- `business_rules.yaml`
- `fpa_config.yaml`
- `fpa_judgement_rules.yaml`
- `domain_context.json`
- `ai_system_prompts_config.yaml`
- 输出模板 manifest

短期强化数据库化：

- 用户、角色、邀请码、登录 session。
- 运行历史。
- 远程任务状态和进度事件。
- 配置变更审计。
- 生成结果审阅记录。
- 项目、工单、上传文件和产物元数据。

不建议立即数据库化：

- FPA 判定规则。
- Prompt 配置。
- 模板 manifest。
- 内置 profile 口径。

## 优先级建议

1. 保持现有配置文件体系，继续完善校验、备份、导入导出和脱敏审计。
2. 将远程任务状态、步骤进度和取消状态补充到 SQLite，避免服务重启后只剩运行历史而缺少过程状态。
3. 将配置变更审计从 JSONL 演进为 SQLite 表，保留现有脱敏规则。
4. 当 Web 端出现项目管理和审阅闭环时，再为项目、工单、审阅记录和人工修改记录设计数据库模型。
5. 只有在规则需要后台化管理、版本发布或项目级绑定时，再考虑把业务规则和 Prompt 纳入数据库版本管理。
