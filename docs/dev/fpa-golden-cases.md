# FPA Golden Case 验收样例

日期：2026-05-29

## 目的

`gen-fpa` 已支持多套 FPA profile。Golden Case 用于人工验收 AI 输出质量，重点检查拆分粒度、类型判断和说明是否符合所选口径。

Golden Case 不追求唯一答案，但输出必须满足对应 profile 的验收标准。

## 通用验收标准

以下标准主要适用于 `current_project`。

### 必须满足

```text
同一三级模块下的界面能力默认合并为 1 条界面开发行。
不按按钮、查询条件、弹窗、状态切换组件拆碎界面行。
非界面逻辑按业务动作拆分。
查询类动作优先判为 EQ。
新增、编辑、删除、维护、保存类动作优先判为 ILF。
导出类动作优先判为 EO。
导入类动作通常判为 EI。
外部接口调用不默认判为 EIF。
只有明确引用外部应用维护的数据组时才可判 EIF。
数据库表、字段、保存表动作不单独生成功能点。
计算依据归类必须来自模板判定原则。
计算依据说明必须对应输入功能过程，不编造不存在的业务。
```

### 不应出现

```text
列表界面开发、查询条件界面开发、新增按钮界面开发、编辑弹窗界面开发被拆成多条。
查询动作被默认判为 ILF。
普通短信、支付、OCR、消息推送等外部服务调用被默认判为 EIF。
新增表、保存字段、维护数据表被单独生成 FPA 行。
同一动作被重复生成多条逻辑处理行。
```

## Case 1：垂直行业管理

### 输入摘要

```text
客户端类型：地市后台
一级模块：垂直行业营销
二级模块：垂直行业管理
三级模块：垂直行业管理
三级模块描述：维护垂直行业基础信息、状态和管理员。

功能过程：
1. 查询垂直行业：按行业名称查询垂直行业列表，支持分页。
2. 添加垂直行业：输入垂直行业名称并保存。
3. 编辑垂直行业：修改垂直行业名称并保存。
4. 删除垂直行业：删除指定垂直行业。
5. 更新垂直行业状态：启用或停用垂直行业。
6. 新增垂直行业管理员：为垂直行业添加管理员账号。
7. 删除垂直行业管理员：移除垂直行业管理员账号。
```

### 期望输出形态

```text
1. 【地市后台】垂直行业营销-垂直行业管理-垂直行业管理-界面开发：EI
2. 查询垂直行业-查询处理开发：EQ
3. 添加垂直行业-逻辑处理开发：ILF
4. 编辑垂直行业-逻辑处理开发：ILF
5. 删除垂直行业-逻辑处理开发：ILF
6. 更新垂直行业状态-逻辑处理开发：ILF
7. 新增垂直行业管理员-逻辑处理开发：ILF
8. 删除垂直行业管理员-逻辑处理开发：ILF
```

### 重点检查

```text
界面行必须合并为 1 条。
管理员弹窗或状态按钮不应拆出独立界面行。
查询必须为 EQ。
添加、编辑、删除、状态更新、管理员维护必须为 ILF。
```

### strict_fpa 期望输出形态

```text
1. 垂直行业信息：ILF
2. 垂直行业管理员关系：ILF
3. 查询垂直行业：EQ
4. 添加垂直行业：EI
5. 编辑垂直行业：EI
6. 删除垂直行业：EI
7. 更新垂直行业状态：EI
8. 新增垂直行业管理员：EI
9. 删除垂直行业管理员：EI
```

重点检查：

```text
不应出现界面开发行。
不应出现逻辑处理开发行。
本系统维护的垂直行业数据组应为 ILF。
垂直行业与管理员之间的关系数据组应为 ILF。
新增、编辑、删除、状态更新等事务功能应为 EI。
```

## Case 2：数据导入模块

### 输入摘要

```text
客户端类型：地市后台
一级模块：客户运营
二级模块：客户数据管理
三级模块：客户名单导入
三级模块描述：运营人员导入客户名单文件，系统校验数据格式并保存有效客户名单。

功能过程：
1. 下载导入模板：下载客户名单导入模板文件。
2. 导入客户名单：上传 Excel 文件，校验手机号、客户类型和归属地，保存有效记录。
3. 查看导入结果：查看成功数量、失败数量和失败原因。
```

### 期望输出形态

```text
1. 【地市后台】客户运营-客户数据管理-客户名单导入-界面开发：EI
2. 下载导入模板-导出处理开发：EO
3. 导入客户名单-导入处理开发：EI
4. 查看导入结果-查询处理开发：EQ
```

### 重点检查

```text
导入客户名单通常判为 EI，因为外部文件数据进入系统并维护内部数据。
下载模板是输出文件，倾向 EO。
查看导入结果是读取查询，倾向 EQ。
不要为“校验手机号”“保存客户名单表”单独生成 FPA 行。
```

### strict_fpa 期望输出形态

```text
1. 客户名单导入：ILF
2. 下载导入模板：EO
3. 导入客户名单：EI
4. 查看导入结果：EQ
```

## Case 3：外部用户中心引用

### 输入摘要

```text
客户端类型：地市后台
一级模块：权限管理
二级模块：账号权限
三级模块：用户中心账号引用
三级模块描述：系统引用统一用户中心维护的人员账号，用于选择业务负责人和查看账号基础信息，本系统不维护账号主数据。

功能过程：
1. 查询用户中心账号：按姓名或手机号查询统一用户中心账号。
2. 选择业务负责人：从用户中心账号中选择负责人并保存到本系统业务对象。
3. 查看负责人详情：查看负责人账号基础信息和所属组织。
```

### 期望输出形态

```text
1. 【地市后台】权限管理-账号权限-用户中心账号引用-界面开发：EI
2. 查询用户中心账号-查询处理开发：EIF 或 EQ
3. 选择业务负责人-逻辑处理开发：ILF
4. 查看负责人详情-查询处理开发：EIF 或 EQ
```

### 重点检查

```text
如输出行明确表示“引用统一用户中心维护的数据组”，可判 EIF。
保存本系统业务对象上的负责人关系，仍应判 ILF。
不要把普通 HTTP 调用四个字机械等同于 EIF。
```

### strict_fpa 期望输出形态

```text
1. 统一用户中心账号：EIF
2. 查询用户中心账号：EQ
3. 选择业务负责人：EI
4. 查看负责人详情：EQ
```

重点检查：

```text
外部维护、本系统引用的统一用户中心账号应为 EIF。
保存本系统业务对象上的负责人选择关系是事务功能，应为 EI。
```

## Case 4：普通外部服务调用

### 输入摘要

```text
客户端类型：地市后台
一级模块：消息管理
二级模块：通知发送
三级模块：短信通知
三级模块描述：运营人员配置短信内容并触发短信发送，系统调用短信平台完成发送。

功能过程：
1. 编辑短信模板：维护短信标题、正文和变量。
2. 发送测试短信：输入手机号发送测试短信。
3. 查看发送记录：查询短信发送状态和失败原因。
```

### 期望输出形态

```text
1. 【地市后台】消息管理-通知发送-短信通知-界面开发：EI
2. 编辑短信模板-逻辑处理开发：ILF
3. 发送测试短信-逻辑处理开发：ILF 或 EI
4. 查看发送记录-查询处理开发：EQ
```

### 重点检查

```text
短信平台是外部服务调用，不应直接判 EIF。
编辑短信模板涉及内部数据维护，应判 ILF。
查看发送记录应判 EQ。
```

### strict_fpa 期望输出形态

```text
1. 短信通知信息：ILF
2. 编辑短信模板：EI
3. 发送测试短信：EI
4. 查看发送记录：EQ
```

重点检查：

```text
短信平台是普通外部服务调用，不应生成 EIF。
```

## Case 5：复杂多界面模块

### 输入摘要

```text
客户端类型：地市后台
一级模块：审批管理
二级模块：客户审批
三级模块：客户准入审批
三级模块描述：包含客户准入申请列表、详情页和审批页，审批页有独立入口和状态流转。

功能过程：
1. 查询申请列表：按客户名称、申请状态查询准入申请。
2. 查看申请详情：进入详情页查看客户资料、附件和历史记录。
3. 审批客户准入：进入审批页填写审批意见并通过或驳回。
4. 导出申请列表：导出当前筛选条件下的申请列表。
```

### 期望输出形态

```text
1. 客户准入申请列表界面开发：EI
2. 客户准入申请详情界面开发：EI
3. 客户准入审批界面开发：EI
4. 查询申请列表-查询处理开发：EQ
5. 查看申请详情-查询处理开发：EQ
6. 审批客户准入-逻辑处理开发：ILF
7. 导出申请列表-导出处理开发：EO
```

### 重点检查

```text
该模块允许多条界面行，但每条界面行必须有 split_reason。
拆分理由应指向独立页面、独立入口或独立业务流程。
不要把列表中的查询条件、按钮、弹窗继续拆成界面行。
```

## 固定样例与自动差异报告

已新增固定 Golden Case fixtures：

```text
tests/fixtures/fpa_golden_cases/vertical_industry_management.json
tests/fixtures/fpa_golden_cases/customer_list_import.json
tests/fixtures/fpa_golden_cases/external_user_center_reference.json
tests/fixtures/fpa_golden_cases/crm_customer_archive_reference.json
tests/fixtures/fpa_golden_cases/erp_order_reference.json
tests/fixtures/fpa_golden_cases/sms_notification_service.json
```

这些文件同时保存：

```text
固定输入 meta。
固定输入功能过程 rows。
current_project 预期 name/type 输出。
strict_fpa 预期 name/type 输出。
```

自动验收测试：

```text
tests/test_fpa_golden_fixture_reports.py
```

执行：

```powershell
.\scripts\test.ps1 tests/test_fpa_golden_fixture_reports.py
```

如果实际输出和预期输出不一致，pytest 会打印差异报告：

```text
FPA Golden Case 差异: <case_id> / <profile>

Expected:
...

Actual:
...
```

后续增加真实项目样例时，只需在以下目录新增 JSON：

```text
tests/fixtures/fpa_golden_cases/
```

测试会自动读取目录中的所有 `*.json` fixture，并分别验证 `current_project` 和 `strict_fpa`。
