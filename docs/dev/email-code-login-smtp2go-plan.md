# 邮箱验证码登录与 SMTP2GO 方案

## 背景

系统需要支持邮箱验证码登录。验证码邮件可通过 SMTP2GO 免费额度先行接入，用于开发测试、内部小范围使用或早期低频场景。

SMTP2GO 免费额度适合起步使用，但不应作为高频生产验证码的长期容量假设。正式使用前必须关注额度、到达率、域名信誉和接口限流。

## 邮箱验证码登录需要的条件

### 邮箱地址可用

用户输入合法邮箱，且系统允许该邮箱登录。

如果是企业系统，通常还需要邮箱已存在于用户表、员工表或白名单中。

### 邮件发送能力

后端需要配置邮件服务，例如 SMTP2GO、企业邮箱 SMTP、Gmail SMTP、Outlook SMTP、SendGrid、阿里云邮件推送等。

使用 SMTP2GO 时，需要：

- SMTP2GO 账号。
- SMTP 用户名和密码。
- 发件邮箱或发件域名验证。
- 后端 SMTP 配置。

### 验证码生成与存储

后端生成一次性验证码，例如 6 位数字。

需要保存：

- 邮箱。
- 验证码哈希。
- 过期时间。
- 使用状态。
- 错误次数。
- 发送频率信息。

验证码可以存放在 Redis 或数据库中。验证码明文不应长期保存。

### 验证码有效期

建议有效期为 5 分钟。过期后必须重新发送。

### 频率限制

必须限制发送频率，避免邮件额度被刷完，也避免验证码接口被滥用。

建议规则：

- 同一邮箱 60 秒内只能发送一次。
- 同一邮箱每天最多发送 10-20 次。
- 同一 IP 每小时最多发送 30-60 次。
- 同一验证码最多允许错误 5 次。

### 验证逻辑

用户提交邮箱和验证码后，后端检查：

- 验证码是否存在。
- 验证码是否匹配。
- 验证码是否过期。
- 验证码是否已使用。
- 错误次数是否超限。
- 邮箱是否仍然允许登录。

### 登录或注册策略

验证码通过后，需要明确系统行为：

- 只允许已有用户登录。
- 不存在则自动注册。
- 不存在则提示无权限。
- 需要绑定员工或组织信息后才能登录。

本项目建议优先采用“只允许已有用户或白名单邮箱登录”，避免验证码登录变成开放注册入口。

### 会话签发

验证通过后，系统需要签发登录态，例如 session、JWT 或 cookie，并更新最后登录时间。

## 免费方案选择

### 现成邮箱 SMTP

可以使用企业邮箱、QQ 邮箱、163、Gmail、Outlook 等 SMTP。

优点：

- 成本最低。
- 开发简单。
- 适合本地开发和临时测试。

缺点：

- 额度和风控不稳定。
- 到达率不可控。
- 不适合正式高频验证码。

### 免费邮件服务商

可考虑：

- SMTP2GO：免费额度适合小系统验证码。
- Brevo：免费额度较大，但平台定位更偏营销邮件。
- Resend：开发体验较好，适合程序发邮件。
- Mailgun：通常偏试用额度，不一定适合长期免费使用。

本项目选用 SMTP2GO。

### 自建 SMTP 邮件服务器

理论上免费，但不推荐。

原因：

- 邮件容易进入垃圾箱。
- 需要维护 SPF、DKIM、DMARC。
- 需要维护 IP 信誉。
- 运维成本高于购买或使用邮件服务。

## SMTP2GO 接入方案

### 推荐方式

优先使用 SMTP2GO 的 SMTP relay。验证码登录场景只需要稳定发送单封邮件，SMTP 接入简单、通用，便于后续替换供应商。

SMTP2GO 也支持 API 发送。如果后续需要发送状态查询、模板化管理或更细粒度的邮件事件追踪，可以再切换或补充 API 方式。

### 环境变量

```env
SMTP_HOST=mail.smtp2go.com
SMTP_PORT=587
SMTP_USERNAME=你的 SMTP2GO SMTP 用户名
SMTP_PASSWORD=你的 SMTP2GO SMTP 密码
SMTP_FROM=noreply@your-domain.com
SMTP_FROM_NAME=系统名称
```

### 外部准备条件

使用 SMTP2GO 前，需要完成：

- 注册 SMTP2GO 账号。
- 添加并验证发件域名，或验证单个发件邮箱。
- 在 SMTP2GO 后台创建 SMTP 用户名和密码。
- 确认发件地址，例如 `noreply@your-domain.com`。
- 测试常见邮箱能否收到邮件。

## 域名验证说明

### 是否需要自己注册域名

如果要验证“发件域名”，通常需要自己拥有一个域名，例如：

```text
your-company.com
```

然后在该域名 DNS 中添加 SMTP2GO 提供的 SPF、DKIM 等记录，证明系统有权限使用这个域名发邮件。

### 推荐：验证自有域名

正式使用时推荐使用自有域名，例如：

```text
noreply@example.com
login@example.com
```

优点：

- 稳定。
- 可信。
- 便于长期使用。
- 邮件品牌和系统身份一致。
- 后续更容易迁移邮件供应商。

缺点：

- 需要购买或使用已有域名。
- 需要配置 DNS。

### 临时方案：验证单个发件邮箱

如果暂时没有域名，可以先用 SMTP2GO 的 single sender email 验证一个现有邮箱，例如：

```text
yourname@qq.com
yourname@outlook.com
yourname@gmail.com
```

适用场景：

- 开发测试。
- 内部小范围验证。
- 快速跑通验证码登录流程。

缺点：

- 到达率和稳定性不如自有域名。
- 发件身份不够统一。
- 不适合作为正式产品长期方案。

### 免费域名方案

存在一些免费域名或免费子域名方案，但不建议用于正式验证码邮件。

#### 免费子域名

示例：

```text
yourname.duckdns.org
yourproject.pages.dev
```

这类通常免费，但很多不适合作为邮件发件域名，因为无法完整控制根域名的邮件 DNS 记录，或者域名信誉较弱。

#### EU.org 免费域名

EU.org 是较老的免费二级域名服务，例如：

```text
yourname.eu.org
```

理论上可以配置 DNS，可能用于验证发件域名。

风险：

- 审批慢。
- 不保证通过。
- 不适合商业正式使用。
- 长期稳定性不如购买域名。

#### Freenom 类免费顶级域名

过去常见 `.tk`、`.ml`、`.ga` 等免费域名，但不推荐。

原因：

- 稳定性差。
- 域名信誉差。
- 很多邮件服务和收件方天然不信任这类域名。

#### Cloudflare Pages 或 GitHub Pages 子域名

这类免费子域名适合托管网站，例如：

```text
xxx.pages.dev
xxx.github.io
```

但不适合作为 SMTP2GO 发件域名。它们主要是网站访问域名，不是可完整控制邮件发信信誉的自有发件域名。

### 域名选择建议

开发测试阶段：

- 不折腾免费域名。
- 直接用 SMTP2GO single sender email 验证现有邮箱。

正式使用阶段：

- 购买或使用已有域名。
- 使用 Cloudflare 免费 DNS。
- 配置 SPF、DKIM、DMARC。
- 使用 `noreply@你的域名` 作为发件地址。

最低成本正式方案：

```text
便宜域名 + Cloudflare 免费 DNS + SMTP2GO 免费邮件额度
```

## 后端接口设计

### 发送验证码

接口：

```http
POST /auth/email-code/send
```

请求：

```json
{
  "email": "user@example.com"
}
```

行为：

- 校验邮箱格式。
- 校验该邮箱是否允许登录。
- 检查同邮箱发送间隔。
- 检查同邮箱每日发送上限。
- 检查同 IP 发送频率。
- 生成 6 位数字验证码。
- 保存验证码哈希和过期时间。
- 通过 SMTP2GO 发送邮件。

### 验证码登录

接口：

```http
POST /auth/email-code/login
```

请求：

```json
{
  "email": "user@example.com",
  "code": "123456"
}
```

行为：

- 校验验证码是否存在。
- 校验验证码是否过期。
- 校验验证码是否匹配。
- 校验错误次数是否超限。
- 验证成功后删除或标记验证码已使用。
- 签发 session、JWT 或 cookie。
- 更新用户最后登录时间。

## 推荐安全参数

```text
验证码长度：6 位数字
验证码有效期：5 分钟
同邮箱发送间隔：60 秒
同邮箱每日发送上限：10-20 次
同 IP 每小时发送上限：30-60 次
验证码最大错误次数：5 次
验证码存储：只存哈希，不存明文
```

## 邮件模板

建议保持短、明确，不加入营销内容。

```text
你的登录验证码是：123456

验证码 5 分钟内有效。若不是你本人操作，请忽略本邮件。
```

## 验证方式

代码实施后，应运行：

```powershell
.\.venv\Scripts\python.exe -m pytest
```

并进行真实邮件链路验证：

- 能收到验证码邮件。
- 错误验证码无法登录。
- 过期验证码无法登录。
- 同邮箱 60 秒内重复发送被拒绝。
- 验证码登录后能进入系统。

## 主要风险

- 免费额度有限，用户量上来后可能不够。
- 邮件到达率依赖域名 DNS 配置。
- SPF、DKIM、DMARC 没配好容易进垃圾箱。
- 验证码接口没有限流时，免费额度容易被刷完。
- 如果允许邮箱不存在时自动注册，可能引入开放注册风险。
