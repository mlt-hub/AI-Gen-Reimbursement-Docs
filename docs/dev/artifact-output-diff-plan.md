# 产出物文档差异比较与统一方案

## 背景

本次分析围绕“产出物文档”进行，目标是明确系统当前定义的交付物、用户指定测试目录中的实际文件、模板契约和生成流程之间的差异，并给出后续统一方案。

分析依据：

- 当前代码仓库中的生成流程与模板文档。
- 内置输出模板及其 manifest。
- 用户指定测试目录 `F:\mlt\mlt-tests\AI-Gen-Reimbursement-Docs\5\`。
- 2026-06-09 在该测试目录内执行的 CLI 运行日志和分步生成结果。

本轮仅整理方案文档，不实施业务代码修改。

## 目标行为

系统应把产出物明确分为三层：

| 层级 | 定义 | 默认展示方式 |
| --- | --- | --- |
| 正式交付物 | 面向客户或送审的最终文档 | 默认展示、下载、计入完整性交付 |
| 审阅附件 | 用于人工审查、质量复核或草稿确认的辅助文件 | 折叠展示，可下载 |
| 过程文件 | Markdown、日志、AI prompt/response、JSON 源数据等调试文件 | 仅在调试或日志入口展示 |

`gen-all` 完成后，应能清晰回答：

- 哪些正式交付物已经生成。
- 哪些正式交付物缺失。
- 缺失是因为生成失败、校验阻断、配置缺失，还是本模式本来不生成。
- 哪些文件只是审阅附件或过程文件，不应被误认为正式交付物。

## 已发现差异

### 1. 交付物数量不一致

系统定义的 `gen-all` 正式交付物应为 4 类：

- FPA 工作量评估。
- 项目功能点拆分表。
- 项目需求清单。
- 项目需求说明书。

测试目录 `F:\mlt\mlt-tests\AI-Gen-Reimbursement-Docs\5\` 当前实际看到：

- 根目录存在 FPA 工作量评估正式 Excel。
- 根目录存在 FPA 审核副本 `*-FPA工作量评估-check.xlsx`，该文件应归类为审阅附件。
- `cosmic文档/` 中存在附件 3 项目需求清单 Excel。
- `cosmic文档/` 中未看到附件 1 项目需求说明书 Word。
- `cosmic文档/` 中未看到附件 2 项目功能点拆分表 Excel。

这说明当前测试目录结果不满足“4 类正式交付物齐全”的预期，后续应通过完整性检查明确标记缺失项，并避免把 FPA 审核副本误计为正式交付物。

### 2. 输出目录命名不准确

当前 pipeline 默认路径为：

```text
FPA 工作量评估      -> 输出根目录
项目功能点拆分表    -> cosmic文档/
项目需求清单        -> cosmic文档/
项目需求说明书      -> cosmic文档/
```

`cosmic文档` 目录实际承载的不只有 COSMIC 文档，也包含需求清单和需求说明书。该命名对用户不够准确，容易让人误解需求说明书和需求清单属于 COSMIC 附属文件。

建议第一阶段先统一 UI、历史、下载包中的展示语义；是否重命名物理目录可放到第二阶段处理。

### 3. 模板与正式产物边界不够清晰

内置 Excel/Word 模板并非空白模板，部分模板中保留示例数据、公式、说明 sheet 或占位符。正式生成时，写入器会在模板基础上删除旧数据、写入新数据、保留公式和样式。

当前机制可用，但用户容易混淆：

- 模板示例内容。
- AI 待填充提示。
- 正式产出内容。

因此需要在文档和 UI 中强调：

```text
输出模板是交付物容器和结构契约，不等同于最终业务数据。
```

### 4. 模板预检只保证结构，不保证正式产物质量

当前 manifest 预检主要检查：

- sheet 是否存在。
- 表头是否存在。
- 数据起始行、样式源行是否存在。
- Word 占位符是否存在。
- 关键公式或固定单元格是否存在。

预检不能保证最终产物数据完整。例如测试目录 5 当前观察到：

- `gen-list` 单步在非交互环境中采用了送审工作量 `0.0`、送审功能点 `0.0` 的默认值，仍写出了附件 3 Excel。
- SPEC 单步已生成 `md\2.1.gen-spec-SPEC-功能需求章节-模板.md`，但在 AI 填充功能过程时超时，未写出附件 1 Word。
- COSMIC 正式 Excel 尚未生成，无法从当前目录判断附件 2 是否满足正式交付质量。

因此应增加“产物后检”，在正式交付物写入后检查关键数据和占位符残留。

### 5. 正式交付物、审阅附件和过程文件混在同一产物语义里

当前 `_artifact(...)` 会登记多类文件：

- 正式交付物。
- FPA 审核副本。
- COSMIC 草稿 Excel。
- COSMIC JSON 草稿。
- COSMIC 校验报告。
- Markdown 中间文件。

这些文件都对开发和审阅有价值，但不应在用户视角被同等称为“交付物”。建议在产物模型中增加 `role`，至少区分：

```text
deliverable
review
debug
```

## 建议方案

### 阶段一：统一产物分类和完整性检查

新增集中式产物定义，建议包含：

```text
kind: fpa | cosmic | list | spec | fpa_check | cosmic_review | log | md
role: deliverable | review | debug
display_name: 用户可见名称
default_relative_path: 默认相对路径
required_in_gen_all: 是否是 gen-all 必需正式交付物
```

目标：

- pipeline、Web 任务、历史记录、下载 ZIP 使用同一套产物分类。
- `gen-all` 完成后可检查正式交付物是否齐全。
- Web UI 默认只展示正式交付物，审阅附件折叠展示，过程文件走日志/调试入口。

### 阶段二：增加正式产物后检

在模板预检之后增加产物后检。

FPA 后检：

- `新增/修改功能点` 不为空。
- `类型` 不为空且在允许值范围内。
- `计算依据归类` 不为空。
- `计算依据说明` 不为空。
- 正式 Excel 不应缺失合计或关键公式。

COSMIC 后检：

- 正式 Excel 写入成功时，关键数据行数量大于 0。
- 不残留 `#AI生成#`。
- 不残留 `${工单内容}` 等变量占位。
- CFP 总和可读取。

需求清单后检：

- 文件存在。
- 项目信息概览和功能清单 sheet 存在。
- 送审工作量、送审功能点与上游 FPA/COSMIC 汇总一致。

需求说明书后检：

- 文件存在。
- 不残留 `{{模块清单表}}`、`{{功能过程详情}}`、`{{文档标题}}` 等关键占位符。
- 如目录未自动更新，提醒前缀规则明确可解释。

### 阶段三：统一目录和下载展示

短期建议：

- 保留物理目录 `cosmic文档/`，避免影响已有路径和测试。
- 在 Web UI、历史页、下载包展示中将其语义改为“正式交付物”或“交付文档”。

中期可评估：

- 将物理目录从 `cosmic文档/` 改为 `交付物/`。
- 如需改名，应同步迁移测试、历史记录、下载打包和文档。

### 阶段四：同步文档

需要更新的文档：

- `README.md`
- `docs/dev/output-template-logic.md`
- 可新增或扩展运行历史/下载相关文档。

文档中应明确：

- `gen-all` 预期生成哪些正式交付物。
- 各单步模式分别生成哪些文件。
- 审阅附件和过程文件不等同于正式交付物。
- COSMIC 校验未通过时，正式 Excel 可能不会生成，只生成草稿或校验报告。

## 风险

### 目录名调整风险

直接把 `cosmic文档/` 改为新目录名，会影响：

- 既有测试。
- 用户已有路径习惯。
- 历史记录。
- 下载 ZIP 内路径。
- 可能存在的外部脚本。

建议先统一展示语义，再评估物理目录改名。

### 后检严格度风险

如果第一版后检过严，可能导致已有可接受产物被阻断。建议将后检分为：

- error：正式交付物不可接受。
- warning：建议人工检查，但不阻断。

第一阶段可先对占位符残留、文件缺失、关键 sheet 缺失设置 error；对部分文本质量问题设置 warning。

### COSMIC 草稿状态风险

COSMIC 当前存在 `passed`、`review_required`、`blocked` 等状态。若 `review_required` 允许写草稿 Excel，必须在 UI 和历史中明确标记为“草稿”，不能误标为正式交付物。

## 决策建议

优先实施顺序：

1. 新增产物角色分类，区分正式交付物、审阅附件和过程文件。
2. 在 `gen-all` 后增加正式交付物完整性检查。
3. 增加正式产物后检，先覆盖占位符残留和关键列为空。
4. 更新 Web/历史/下载展示，避免把草稿和调试文件当正式交付物。
5. 同步 README 和输出模板文档。
6. 最后再评估是否重命名 `cosmic文档/` 物理目录。

第一阶段验收标准：

- 用户能清楚看到 4 类正式交付物状态。
- 项目需求清单缺失时，不会被误判为完整交付。
- COSMIC 草稿不会被误判为正式项目功能点拆分表。
- 指定测试目录生成后，附件 1/2/3 文件名与输入 Excel 元数据一致。
