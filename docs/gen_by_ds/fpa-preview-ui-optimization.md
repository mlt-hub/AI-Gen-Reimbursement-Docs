# FPA 预览页界面优化方案

## 背景

FPA 预览页用于在生成完整文档前，按三级模块查看生成出的 FPA 行，确认新增或修改的功能点、类型、计算依据和生成方式。

当前页面已经具备模块选择、FPA 方案选择、执行策略选择、结果表格、审核信息和说明详情。后续优化重点是降低审阅成本，让用户能按“业务条目 -> 类型 -> 依据”的顺序快速扫表。

术语以《生成结果审阅术语规范》为准：[`result-review-terminology.md`](./result-review-terminology.md)。

FPA Prompt 来源、配置缺失处理和代码内置 Prompt 兜底策略以 [`fpa-prompt-configuration.md`](./fpa-prompt-configuration.md) 为准。

---

## 当前问题

### 1. 表格阅读顺序不够自然

当前表头顺序为：

```text
# / 类型 / 功能点 / 归类 / 方式
```

用户审阅时第一关注点通常是“新增或修改了哪个功能点”，然后才判断类型是否正确。因此 `类型` 放在 `功能点` 前面，会让阅读焦点先落到分类结果，而不是业务对象。

### 2. 术语需要统一

当前页面存在 `功能点`、`归类`、`方式`、`说明详情` 等表述。根据用户指定规范，应统一为：

```text
新增/修改功能点 / 类型 / 生成方式 / 计算依据归类 / 计算依据说明
```

### 3. 说明信息离表格行较远

当前 `说明详情` 作为整体折叠块位于表格下方。用户审阅某一行时，需要在表格行和底部说明之间来回跳转。

更好的方式是让每一行可以在同一上下文中看到 `计算依据说明`，保持“这一行是什么、为什么这样归类、详细依据是什么”的连续阅读。

### 4. 模块选择未来可能不适合长下拉

当前三级模块使用下拉框选择。模块数量少时可接受；模块数量较多时，长下拉会降低定位效率，也不利于比较多个模块。

## 第一阶段优化

第一阶段只调整 FPA 预览结果区，不大改页面整体布局。

### 结果区标题

结果区顶部应增加明确标题：

```text
FPA 功能点估算
```

推荐层级：

```text
FPA 功能点估算
三级模块名称
N 个功能过程 · FPA 方案 · 执行策略                    AI / 兜底
```

`FPA 功能点估算` 用于说明当前结果区的业务内容；三级模块名称用于说明当前审阅对象。两者不要合并，避免页面标题和业务对象混在一起。

### 表格列顺序

推荐改为：

```text
# / 新增/修改功能点 / 类型 / 生成方式 / 计算依据归类
```

字段映射：

| 表头 | 数据字段 | 展示建议 |
|---|---|---|
| # | 行号 | 固定窄列 |
| 新增/修改功能点 | `row.name` | 主文本，加粗显示 |
| 类型 | `row.type` | 使用紧凑 badge，保持醒目但不抢占首列焦点 |
| 生成方式 | `row.generation` | 窄列展示规则、AI、兜底等来源 |
| 计算依据归类 | `row.classification_basis` | 展示短依据，空值显示 `-` |

### 行级说明

把 `row.explanation` 统一命名为：

```text
计算依据说明
```

推荐展示方式：

- 每行下方展示一段简短说明，或
- 每行提供展开/收起区域，只展开当前行的计算依据说明。

如果说明内容较长，优先使用行级展开，而不是页面底部统一折叠块。

### 审核信息

`审核信息` 可以保留，继续用于展示覆盖率、规则集、版本、缺失功能过程和生成方式统计。

注意：审核信息是模块级汇总，不应承载单行的 `计算依据说明`。

### AI 调试信息

FPA 预览页应在结果区下方提供默认折叠的 `AI 调试信息`，用于排查 FPA 行生成质量、Prompt 配置和响应解析问题。

推荐区块：

| 区块 | 内容 | 展示规则 |
|---|---|---|
| 系统提示词 | 发给模型的 system prompt | AI 实际调用后展示 |
| 用户提示词 | 发给模型的 user prompt | AI 实际调用后展示 |
| AI Prompts | system 与 user 的完整组合视图 | 便于整体复制和人工审阅 |
| AI Responses | 模型返回的原始文本 | JSON 解析前的原始响应 |
| AI Thinking | 模型返回的 thinking 内容 | 仅当 API 实际返回时展示；不伪造、不推断 |
| 解析结果 | 从原始响应解析出的 rows | 便于对照 AI 原始输出和页面最终结果 |

规则优先、仅规则或 AI 调用失败后兜底时，也应展示调试区，但标记为未实际使用 AI 或展示失败原因。

安全边界：

- 不展示 API Key、请求头、环境变量或本机配置路径。
- `AI Thinking` 只展示模型实际返回内容；没有返回时显示“当前模型未返回思考过程”。
- 调试区默认折叠，不干扰普通 FPA 审阅。
- 调试信息只用于 Web 预览，不写入正式交付文档。

## 第二阶段优化

当三级模块数量较多时，可以把顶部下拉升级为“模块列表 + 结果区”的结构。

推荐方向：

- 左侧为可搜索三级模块列表。
- 右侧为当前模块的 FPA 预览结果。
- 模块列表显示模块名、功能过程数量、是否已有预览结果。
- 点击模块后右侧直接刷新预览上下文。

该阶段适合在模块数量变多、用户需要频繁切换模块时实施。

## COSMIC 复用方向

COSMIC 后续如果也需要预览和人工审阅，应复用同一套生成结果审阅结构。

可参考映射：

| 抽象字段 | FPA 显示 | COSMIC 可显示 |
|---|---|---|
| 结果项 | 新增/修改功能点 | 新增/修改功能过程 |
| 类型 | 类型 | 数据移动类型 |
| 归类依据 | 计算依据归类 | 计算依据归类 |
| 说明依据 | 计算依据说明 | 计算依据说明 |
| 生成来源 | 生成方式 | 生成方式 |

如果 COSMIC 标准术语要求不同，应先更新 [`result-review-terminology.md`](./result-review-terminology.md) 的映射，再实施页面。

---

## 验收标准

- 结果区顶部显示 `FPA 功能点估算`。
- `FPA 功能点估算` 与三级模块名称分层展示，不把模块名称作为结果区唯一标题。
- FPA 表格列顺序为 `# / 新增/修改功能点 / 类型 / 生成方式 / 计算依据归类`。
- 页面不再使用 `功能点`、`归类`、`方式`、`说明详情` 表达对应字段。
- `计算依据说明` 与具体行保持上下文关联。
- 模块级 `审核信息` 与行级 `计算依据说明` 分工清晰。
- 结果区提供默认折叠的 `AI 调试信息`，可查看系统提示词、用户提示词、AI Prompts、AI Responses、AI Thinking 和解析结果。
- `AI 调试信息` 不展示 API Key 或其他敏感配置。
- 后续 COSMIC 页面可以按同一结构复用，不需要重新设计审阅术语体系。
